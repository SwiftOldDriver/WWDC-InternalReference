## WWDC 21 新特性与痛点优化

WWDC 21 苹果针对该 Session，主要为我们带来了以下四类后台 API：

* Invoice lookup API
* Refunded purchases API
* Renewal extension API
* Consumption API

两类 StoreKit 相关的 API：

* Manage subscriptions API
* Request refund API

本节会结合现状痛点，分别对上述新特性进行逐一解析



### Invoice lookup API —— 苹果订单关联能力优化，可用于搭建客诉平台

长期以来，IAP 的客诉处理一直一言难尽。**其根本原因在于没有办法将用户支付后获得的苹果订单与业务的应用订单关联起来**。主要痛点表现为以下两方面：

- 无法判断用户提供的苹果订单截图的真实性：我们团队有幸见识到 PhotoShop 高手高仿订单截图，从那以后，我们遇到类似的场景，只能引导用户联系苹果进行退款处理。而苹果的退款流程也是一言难尽，大大降低了用户体验充值的快乐，同时也误伤了一部分真实用户（不允许退款，但确实未发货的用户）。
- 无法判断用户提供的苹果订单是否已经发货：由于用户提供的只有苹果订单相关的信息，我们没有办法利用苹果订单反查出业务订单。所以灰产很有可能多次利用同一截图，进行骗补的操作

WWDC 2021，苹果终于提出让人期待已久的 Invoice lookup API，我们未来可以直接通过用户反馈的苹果订单，关联到具体的业务订单，大概的流程如下：

1、引导用户提供苹果订单截图 / 直接引导用户提供苹果订单中的 Invoice Order ID（如下图）

![img](https://cdn.nlark.com/yuque/0/2021/png/496659/1624729312042-2c0683b3-ed5e-4e18-8282-eb061bcf5cd4.png)

（若用户提供是苹果订单截图，我们可以通过图像识别 / 人工客服提取出 Invoice Order ID）

2、我们可以通过 Invoice lookup API 进行订单问题定位，其返回的 JWS 形式的票据内容

3、我们可以通过解码 JWS 形式的票据，获得苹果唯一订单标示，继而关联至业务订单

4、最终可实现对用户进行查单 / 智能补发等操作



Invoice lookup API 需要提供 {customer_order_id}（也即上文中搜集的 Invoice Order ID）与 {appAppleId}

- API：/inApps/v1/lookup/{customer_order_id}

### Refunded purchases API —— 对历史退款发放的商品进行索回

一直以来，IAP 一直存在退款灰产。在 WWDC 2020 年之前，用户退款是没有退款通知的，这也就是说开发者对用户退款这件事情无感。但是问题来了，我们的货早已在支付的时候发出，如果我们不知道用户退款了，又怎么样对发出的货进行回收呢？答案是没办法回收，这也就导致了货发了，钱没收到。由此也产生了一条灰产链：**灰产使用自身苹果 AppleID 给其他用户进行代充，代充完毕后再前往苹果进行退款；或直接提供代退款服务**。苹果为了打击这条链路，每年都对用户退款进行收紧。同时终于在去年正式发布退款通知，我们才能终于感知用户退款这一事件，继续对已发的货进行索回。

当前苹果提供的退款相关能力只有退款通知（REFUND），基本上能够覆盖风控索回、数据分析等场景。但是我们作为支付平台，在推进业务接入退款通知时，往往会遇到以下问题：

- 不少业务已经上线很多年，甚至开发团队已经解散，只有外包团队在维护。这部分团队不会接入也不想接入退款通知，但是由于恶意退款，仍旧对业务造成了不少的损失
- 业务从上线到退款通知接入，往往有一个比较长的间隔。同时部分已上线的业务，也希望对退款通知接入前的退款历史订单进行索回

WWDC 2021，苹果为我们提供了 Refunded purchases API，用于主动判断某笔订单是否退款。**有了该接口，我们就可以起一个任务，遍历所有的历史订单，对未接入退款通知 / 接入退款通知前的退款订单进行索回**



Refunded purchases API 需要提供订单唯一标示 {original_transaction_id} 与 {appAppleId}

- API：/inApps/v1/refund/lookup/{original_transaction_id}

### Renewal extension API —— 赠送一定的包月时长，用于挽留用户

众所周知，因为一些原因，NBA 停止在中国播放了一段时间。但在 NBA 停播之前，不少用户为了观赛购买了一些体育类应用的会员。现在 NBA 停赛了，业务希望补偿用户，赠送用户一定的包月时长。那么此时业务只能搜集这批用户，在业务数据库进行时长增加的操作。

但这会产生一个问题，假设用户之前开通的是连续包月（对应苹果物品自动续期订阅类型），哪怕这个业务给用户赠送了两个月的时长，那么根据 IAP 的规则，他仍旧会每月扣费。这就导致一个问题，用户永远有多出的两个月时长，而且没办法进行消耗。出现这种状况，用户只能退订（取消自动续期订阅），才能够把赠送的两个月时长给消耗掉。

但新的问题又来了，我们作为开发者，肯定不希望用户关闭连续包月。更何况用户要是连续一年购买同一订阅组的自动续期订阅，我们还能享受 85% 的分成（对比普通购买仅能享受 70% 的分成）。但这种场景，目前确实无解，只能让用户取消断档，才能处理用户的投诉。

终于在 WWDC 2021，苹果为我们提供了新的思路。苹果为我们新增了一个 Renewal extension API，用于免费延长自动续期订阅的时长。通过该接口延长续费时长，能够解决我们上述场景中遇到的问题，即在用户**不断档（取消自动续期订阅）**的情况下，赠送用户**可消耗完**的订阅时长。

**同时，该接口也可以用于用户促销（可以结合不同业务的场景，免费给用户赠送一定的订阅时长）**



Renewal extension API 需要提供订单唯一标示 {original_transaction_id} 、延长时间 {extendByDays} 与 延长原因 {extendReasonCode}

- API：/inApps/v1/subscription/extend/{original_transaction_id}
- 调用结果如下图示：

![img](https://cdn.nlark.com/yuque/0/2021/png/496659/1624729540674-1d8f7047-ae45-492b-aee1-ee894eaa8333.png)

需要注意的是，该接口有以下限制：

- 每次最多延长 90 天
- 一年针对具体某个订阅，最多延长两次

- 延长的时间不算在 85% 分成的一年时长



### Consumption API —— 退款前置通知（已上线）

一直以来，用户恶意退款都是重要的 IAP 灰产获利方式之一。好消息是，苹果也发现了这个问题，并一直不断在优化关于退款处理的逻辑。从一开始的开发者无感知，到上一年新特性退款通知，再到今年的新特性，都在不断降低恶意退款的占比。

上一年发布的退款通知虽然可以让开发者感知用户退款事件以进行索回，但仍存在一个灰产场景。如果用户购买的是应用内的货币，然后将其立即消耗完，此时我们是没办法做有效的索回的。

典型的场景是，灰产通过 AppleID 购买王者荣耀点券，然后于淘宝 / 拼多多平台上销售皮肤，等找到买家后，通过赠送的手段，使用点券赠送皮肤至买家，并将账号上剩余点券消耗完毕。此时，灰产再通过苹果退款，获取退款款项，我们也收到了苹果退款通知。但因为灰产的账号点券余额已为零，我们没办法对其进行有效索回，只能将其扣为负值，待其充值后再进行索回。但这又引发了一个问题，灰产使用的游戏内账号，大部分都是通过养号获得的一次性账号，他不关心该账号会不会被扣到负值。哪怕该账号被封了也无所谓，毕竟灰产下次也是用其他王者账号进行充值购买操作。

类似的问题，在其他不同场景下也有很多类似案例。关键原因在于退款审核是由苹果处理的，但苹果并没有开发者侧的账号信息，所以苹果只能单纯根据当前 AppleID 的风控情况进行退款审核判断。

为了解决该问题，苹果在 WWDC 2021 新增了 Consumption API。需要注意的是，这里提到的 Consumption API 包含两个含义：

- 狭义的 Consumption API：我们通过 API 把业务侧信息告知苹果的接口
- 广义的 Consumption API：既包含狭义的 Consumption API，还包括 CONSUMPTION_REQUEST 通知

苹果目前在文档中使用的定义是广义的 Consumption API，即包含了 CONSUMPTION_REQUEST 通知，所以本文之后的 Consumption API 含义与苹果保持一致，即指广义的 Consumption API



整个 Consumption API 实际上就是为我们提供了一个退款前置查询的能力，其整个交互过程如下：

1、用户退款

2、苹果收到退款请求后，会**在 48 小时内**进行审核，同时发送 CONSUMPTION_REQUEST 通知至开发者

3、开发者收到 CONSUMPTION_REQUEST 通知后，**需要在 12 小时内**，调用狭义的 Consumption API 进行请求，告知苹果用户信息

4、苹果根据开发者反馈的信息，结合 AppleID 信息，对退款用户进行审核**（需要注意的是，苹果不会完全采信开发者信息，仅用做参考）**

5、苹果同意退款

6、开发者根据物品类型差异，收到退款通知



狭义的 Consumption API 同样需要提供订单唯一标示 {original_transaction_id}  以及一些列用户状态相关的信

- API：[/inApps/v1/transactions/consumption/{original_transaction_id}](https://developer.apple.com/documentation/appstoreserverapi/send_consumption_information)
- 调用结果如下图示：

![img](https://cdn.nlark.com/yuque/0/2021/png/496659/1624729418974-61e5f561-d555-432b-a9dc-b6fca71ded78.png)

需要注意以下几点：

- **由于交互过程中涉及到将在某应用的用户信息传输至苹果侧，所以需要在用户隐私协议中予以声明**
- 目前必传字段仅为 {customerConstentd}、{consumptionStatus}，其他字段可以根据用户授权情况判断是否传输至苹果侧。

- 根据我们调研，目前该接口仍存在一个时间差问题：由于开发者通知苹果和苹果做出退款同意决定仍有一定时间差，如果开发者通知苹果用户尚未消耗游戏币，但过了一小时后，用户又对游戏币进行消耗了，那么相当于我们给苹果的信息已经是"过期的"，这样此时苹果由于开发者反馈游戏币尚未消耗，所以同意退款，导致仍旧出现历史问题
	- 关于这个问题，业务目前可以考虑在收到 CONSUMPTION_REQUEST 通知时，对账户进行冻结处理（48 小时后调用 Refunded purchases API 查询订单状态，若未退款，予以解冻）



### Manage subscriptions API —— 应用内拉起自动续期订阅管理界面（StoreKit 2, iOS 15）

众所周知，自动续期订阅管理界面入口藏的十分深，主要原因在于无法在应用内直接拉起自动续期订阅管理界面。用户需要切换至系统的设置 App，然后点击 AppleID，然后点击订阅，然后才能选择具体应用，最后才能拉起订阅相关界面。这导致相当多的用户没有办法关闭自动续期订阅，甚至搜索引擎上还出现"教你如何关闭苹果续费"等知识问答

WWDC 21，苹果为了解决该问题，为我们提供了应用内拉起自动续期订阅管理界面的 StoreKit 接口，我们只需要传入 UIWindowScene 对象即可

* API：[showManageSubscriptions(in:)](https://developer.apple.com/documentation/storekit/appstore/3803198-showmanagesubscriptions?language=objc)
* 调用结果如下图所示：

![img](https://cdn.nlark.com/yuque/0/2021/png/496659/1624728287391-e0f2beb6-0fca-4c1e-bf2a-48d710293a87.png)

* 该接口接入后，可能会引起订阅切换 / 订阅取消曲线上涨
* 该接口可能会成为未来的审核标准之一



### Request refund API —— 应用内退款（StoreKit 2, iOS 15）

同样的，搜索引擎上也存在着大量"如何在苹果系统上退款"的知识问答。出现这个情况的原因也是在于退款入口藏的十分之深，同时苹果亦无提供相关接口供开发者调用

WWDC 21，苹果为了解决问题，为我们提供了应用内退款接口，我们只需传入需要退款的 transactionID 及 UIWindowScene 对象即可

* API：[beginRefundRequest(for:in:)](https://developer.apple.com/documentation/storekit/transaction/3803219-beginrefundrequest)

- 调用结果如下图所示：

![img](https://cdn.nlark.com/yuque/0/2021/png/496659/1624728315292-a6522c5d-d7e0-4acf-ae10-d2cd643ce8a1.png)

需要注意的是：

* 通过该接口的调用，开发者服务器会收到 REFUND / REFUND_DECLINED 的通知
	* **目前非该接口调用引起的退款，不会收到 REFUND_DECLINED 通知**
* 该接口可在沙箱测试
* 该接口接入后可能会引起退款曲线上涨
* 该接口可能会成为未来的审核标准之一
* 该接口有可能被灰产攻击

## 结语

总的来说，在我看来，苹果这次算是给了 IAP 开发者一个大大的惊喜。针对许多反馈已久，甚至已不抱希望的问题，苹果终于提供了其优化方案与思路。未来我们可以基于 Invoice lookup API 搭建客诉平台，解决支付问题难排查的痛点；可以基于 Refunded purchases API 与 Consumption API 优化灰产恶意退款的场景，一定程度上给予灰产打击并降低业务坏账比例；可以基于 Renewal extension API 做自动续期订阅的促销与挽回

以下为现状与 WWDC 21 新特性的简要对比，读者可用作速查参考
