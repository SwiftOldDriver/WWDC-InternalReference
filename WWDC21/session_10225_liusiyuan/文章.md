# WWDC21 10225 - 使用Group Activity共享媒体

[![RiU3kD.png](https://z3.ax1x.com/2021/06/20/RiU3kD.png)](https://imgtu.com/i/RiU3kD)

## 认识SharePlay
疫情逐渐褪去后的一届WWDC，Apple也推出了一个符合时宜的新功能SharePlay。使用苹果设备的用户，无论是iOS、iPadOS还是MacOS，都可以通过FaceTime开启SharePlay，和其他苹果用户，无论他使用什么类型的苹果设备，都可以实时的共享电影、音乐等媒体。SharePlay的关键词有两个：share 和 play，分享和播放。当然这里播放的语意可以延伸，不仅仅是媒体的播放，更像是一种屏幕共享。

让我们先看看当下是如何处理这两件事情的。当我们想要视频时，往往是通过微信、或者工作使用的IM；当我们想要分享时，往往是在产生内容的程序中，通过分享链接到第三方IM中，接收方打开再跳转到对应的App。
这中间可能存在了两个问题：第一是整个分享的链路很；第二是整个过程并非实时，在某些特定的场景下，也增加了接收信息的用户打开并接受这条分享的成本，导致分享的行为不能顺利完成。

SharePlay提供了一种新的思路，通过FaceTime直接发起视频，在视频过程中分享媒体，FaceTime的双方都可以对该媒体进行控制。系统层级的API直接减少了链路，实时性的分享也满足了不少特殊场景下的使用，比如疫情下的聚会，联络，甚至是教学、工作等。

FaceTime在国内的存在感比较低，当我们拿到一个新的苹果设备，这个应用可能的归宿几乎是三个：删除、移到资源库或者移到一个永运不会打开的文件夹内，里面包含了所有系统附赠的但不曾打开的程序。不过好的工具恰巧需要无感知性，当无感知的FaceTime和SharePlay和高粘性多样性的App结合起来时，或许会产生新的魔法效果。

## 关于本文
本文侧重SharePlay的具体实现。使用groupActivity的为你的程序增加SharePlay的功能，用画中画模式来优化SharePlay的体验，使用AVFoundation新增的`AVPlaybackCoordinator`让多个设备的媒体同步。

SharePlay在不同端的设备上都可用，另一篇文章【占位符/文章发出后补充】会描述如何在safari接受其他用户从iOS App上发起的SharePlay。

## 开始SharePlay
首先我们看看一次SharePlay的流程

[![RiUn61.png](https://z3.ax1x.com/2021/06/20/RiUn61.png)](https://imgtu.com/i/RiUn61)

当两台设备在同一个FaceTime Call中，其中一台打开一个可以进行SharePlay的程序，并发起SharePlay时，GroupActivity框架会初始化一个GroupSession，同时生成对应的GroupActivity。同时，这个session也会分享给第二台设备，第二台设备首先会唤起对应的App，session也会创建一个GroupActivity的实例。到此为止，两台设备在同一个session中，可以进行接下来的活动。

### 创建GroupActivity
要使我们的App拥有SharePlay的能力，首先我们需要定义一个用于share的group activity。注意，group activity的API目前只支持swift 5.5（又一个转swift的理由）。下面是一个示例程序：

```swift
import GroupActivities

// A type that represents a movie to watch with others.
struct Movie: Hashable, Codable {
    var url: URL
    var title: String
    var description: String
    var posterTime: TimeInterval
}

// A group activity to watch a movie together.
struct MovieWatchingActivity: GroupActivity {
    
    // The movie to watch.
    let movie: Movie
    
    // Metadata that the system displays to participants.
    var metadata: GroupActivityMetadata {
        // implementation
    }
}
```

首先引入新的GroupActivities，我们需要实现的协议是`GroupActivity`，其中主要有两个属性值得注意：一个是activityIdentifier，这是这个活动的唯一标志符。同一个App可能会定义多个用于SharePlay的活动，我们可以用这个属性来区分不同的活动。如果开发者没有自己实现，协议默认使用App的bundle identifier + 当前活动的类名或结构体名。以上面的代码为例，这个MovieWatchingActivity的标志符就是`com.mycompany.App. MovieWatchingActivity`。另一个属性是GroupActivityMetadata，其中包含了该活动需要展示的UI信息，比如title、previewImage等。metadata同样会定义这个活动的类型，目前支持的枚举有 listenTogether、watchTogether和默认的generic。可以看到目前Apple对SharePlay的推荐使用场景还是媒体的共享。

GroupActivity中包含的还有真正需要共享的数据，在示例中是movie这个结构。可以看到它遵循codable，其中包含的内容很简单，一些展示信息和代表媒体的URL。Apple不建议共享的数据太庞大。在用户发起FaceTime Call的时候，如果同时发起了SharePlay，要共享的activity就会被携带发送到接收方用户的设备上，遵循codable协议也是为了便于解析，因此数据应只包含必须有的信息，越简单越好。

### 发起流程
发起方需要决定是否分享某个活动，这里可以调用`prepareForActivation`。这个方法的输出是三个枚举：
`activationDisabled `用于用户不在FaceTime Call中或者不想进行分享；`activationPreferred `用于用户在FaceTime Call中并且也愿意分享；`cancelled `用于取消。

```swift
// Create a new activity for the selected movie.
let activity = MovieWatchingActivity(movie: selectedMovie)

// Await the result of the preparation call.
switch await activity.prepareForActivation() {
      
    case .activationDisabled:
        // Playback coordination isn’t active, or the user prefers to play the
        // movie apart from the group. Enqueue the movie for local playback only.
        self.enqueuedMovie = selectedMovie
            
    case .activationPreferred:
         // The user prefers to share this activity with the group.
         // The App enqueues the movie for playback when the activity starts.
         activity.activate()
            
    case .cancelled:
        // The user cancels the operation. Do nothing.
        break
            
    default: ()
}
```

### 通过GroupSession管理共享
GroupActivity的生命周期并非由App所管理，而是通过GroupSession。下图可以展示一个GroupSession的生命周期：两台设备会接收到GroupSession，系统会打开对应的App准备好状态，就绪后在加入到GroupSession中。

[![RiUMm6.png](https://z3.ax1x.com/2021/06/20/RiUMm6.png)](https://imgtu.com/i/RiUMm6)

GroupSession是真正管理整个共享过程的对象。它提供了目前正在激活的GroupActivity、活跃的用户、和连接状态，以及共享播控状态。App不能主动创建GroupSession，获取到GroupSession的唯一方法是通过GroupSession Async Sequence，不管是发起方还是接收方都要从这个途径接受下发。当一个GroupActivity被激活时，系统会创建GroupSession，并给到对应的await循环中。

```swift
for await groupSession in MovieWatchingActivity.sessions() {

}
```

### 通过`AVPlaybackCoordinater`共享状态
GroupSession建立之后，需要同步不同设备之间的播放状态。这里使用AVFoundation的新API：`AVPlaybackCoordinater`。这一部分我们会在后面详细介绍，目前先走完SharePlay的流程。操作很简单，只需要将session挂载到AVPlayer的Coordinater上即可，框架会同步好播控状态。

```swift
private let player = AVPlayer()

// The group session to coordinate playback with.
private var groupSession: GroupSession<MovieWatchingActivity>? {
    didSet {
        guard let session = groupSession else {
            // Stop playback if a session terminates.
            player.rate = 0
            return
        }
        // Coordinate playback with the active session.
        player.playbackCoordinator.coordinateWithSession(session)
    }
}
```

### 加入和结束session
在两端App加入到session之前，session处于wait的状态。当App就绪时，可以调用join方法加入到session中。同时因为共享的内容和状态随时会变化，因此需要监听session的状态和真正用于共享的内容。

```swift
for await groupSession in MovieWatchingActivity.sessions() {
    // Set the App's active group session.
    self.groupSession = groupSession
    
    // Remove previous subscriptions.
    subscriptions.removeAll()
    
    // Observe changes to the session state.
    groupSession.$state.sink { [weak self] state in
        if case .invalidated = state {
            // Set the groupSession to nil to publish
            // the invalidated session state.
            self?.groupSession = nil
            self?.subscriptions.removeAll()
        }
    }.store(in: &subscriptions)
    
    // Join the session to participate in playback coordination.
    groupSession.join()
    
    // Observe when the local user or a remote participant starts an activity.
    groupSession.$activity.sink { [weak self] activity in
        // Set the movie to enqueue it in the player.
        self?.enqueuedMovie = activity.movie
    }.store(in: &subscriptions)
}
```

结束session有两个方法：`end()`和`leave()`。前者会终止所有用户的SharePlay，后者仅会终止自己的。

🆒 到此为止，一个可以用来SharePlay的App就完成了！


## 使用画中画来增强体验
支持画中画模式可以带来的好处就是：直接启播，用户在接受SharePlay时不需要额外的操作。这极大的减少了用户操作的链路，体验上无缝衔接，纵享丝滑。

GroupSession会在后台指示App准备播放状态，同时不进入全屏打开，直接进入正常的流程，在准备就绪后直接启播。

但是也有一些问题会导致无法启播，比如App需要用户登陆、或者资源无法播放，在这些情况下，GroupSession 提供了API，使App可以弹窗提醒用户进入前台操作。

更多关于如果增强SharePlay体验的tips，请见[Design for Group Activities](https://developer.apple.com/videos/play/wwdc2021/10184/)

## 共享状态如何完成：`AVPlaybackCoordinator`
SharePlay的播放状态是两方都可以进行控制的，并且进度会实时的更新到设备上。这就要归功于`AVPlaybackCoordinator`。这是AVFoundation在iOS 15更新的新API。我们来简单的认识一下它：

[![RiUMm6.png](https://z3.ax1x.com/2021/06/20/RiUMm6.png)](https://imgtu.com/i/RiUMm6)

`AVPlaybackCoordinator`有两个子类：`AVPlayerPlaybackCoordinator `总是会挂载到一个AVPlayer上，并且会处理所有的播放变化，这个是较为建议使用的一个类；`AVDelegatingPlaybackCoordinator `提供更灵活的方法，可以用于非AVPlayer的情况。本文主要针对前一种的工作原理进行描述。

### `AVPlayerPlaybackCoordinator `如何工作
那么`AVPlayerPlaybackCoordinator `究竟是如何工作的呢，可以看下图：

[![RiUlTO.png](https://z3.ax1x.com/2021/06/20/RiUlTO.png)](https://imgtu.com/i/RiUlTO)
 
前文中我们使用`player.playbackCoordinator.coordinateWithSession(session)`将`AVPlayerPlaybackCoordinator `注册到session中。完成这一步后，两个设备的AVPlayer就连接了起来。`AVPlayerPlaybackCoordinator `会拦截所有AVPlayer内部的状态变化，将这个变化传递给另一台设备的`AVPlayerPlaybackCoordinator `同时暂缓自己设备的状态更新，在双方都获取到新的状态后，双方的`AVPlayerPlaybackCoordinator `会选取一个合适的时机，更新自己设备的AVPlayer。

举个🌰，两台设备Alice和Bob在共享一个视频。Alice拖动了进度条，进行了seek操作。她的`AVPlayerPlaybackCoordinator `会拦截到这个操作，并且在AVPlayer进行seek后，进入到wait的timeControlStatus中。同时，`AVPlayerPlaybackCoordinator `会将这个seek指令发送给Bob设备上的`AVPlayerPlaybackCoordinator `，Bob的AVPlayer会执行这个指令，并且也进入wait状态。当所有人就绪后，继续进行播放。

### 正常工作的前提：同一item & time match
这里有两点需要注意：
- 两个播放器必须播放同一个item
- 两个播放器的时间必须是吻合的

逐条解释，首先看第一条。`AVPlayerPlaybackCoordinator `共享生效的前提是：双方的AVPlayer中在播放同一个item。也就是说，在同步状态之前，必须将要共享的item设置为AVPlayer的currentItem。否则所有的播控是无效的。加入到共享的item会以目前已经在session中的item的状态为准。

还是以Alice和Bob为例。假设Alice已经在进行共享，并且播放了5s的视频。Bob由于网络问题才刚刚加入，此时Bob的currentItem会以已经在session中的Alice的状态为准，也就是会从5s处开始播放。而在Bob加入之前，Alice的操作并不会对Bob有影响。当然了，在这个case中，Bob可能就错过了一些精彩瞬间，后面我们会通过另一种方式来增强Bob的体验。

那么我们如何判断两个item是不是同一个，还记得前面movie结构体定义的URL吗？这是一个方法，直接判断AVAsset的URL是否一致。这个方法的问题是，有些App提供了缓存，如果一方使用缓存URL，另一方使用server的URL依旧是不行的。在这种case，开发者可以使用下面的方法，给每个AVPlayerItem进行标识。如果这个方法被实现，`AVPlayerPlaybackCoordinator `会以标志符为准。

```swift
@protocol `AVPlayerPlaybackCoordinator `Delegate <NSObject>

-(NSString *)playbackCoordinator:(`AVPlayerPlaybackCoordinator ` *)coordinator identifierForPlayerItem:(AVPlayerItem *)playerItem;
```

关于第二条，我们假设一个场景：Alice和Bob同时在看一个视频。这时Alice的视频中插入了一个广告，但Bob并没有。假设广告的播放器和视频的播放器是一个，那么就会导致视频的时间不一致，共享就会失效。合理的做法是，采用两个播放器，当广告插入的时候，Alice切换到另一个播放器观看广告，广告结束之后切换到视频播放器，`AVPlayerPlaybackCoordinator `获取到session的进度，同步给Alice。

### 不理想的情况：播放中止的处理
前文中提到了许多不理想的情况：Bob的网络比较慢，Alice被迫观看广告等。还有一些情况会导致播放中断，比如来电话，闹钟响了等被迫暂停AVPlayer。我们并不希望这种暂停并影响其他用户，也不希望被终止的用户错过精彩内容。这里可以使用AVCoordinatedPlaybackSuspension。

AVCoordinatedPlaybackSuspension可以使某个用户的播放状态隔离于整个session。在这个状态下的所有操作不会影响到别人。AVCoordinatedPlaybackSuspension有两种，`AVPlayerPlaybackCoordinator `会在需要的时机自己添加。比如AVAudioSession被迫中断，或者一个用户在seek过程中拖动进度条时的操作。当这些行为结束后，这个用户会结束隔离，重新和session同步。

当然了，开发者可以使用这个特性。我们假设一个场景：Alice和Bob在看电影，Bob的闹钟突然响了，在结束闹钟的时间里，Bob错过了一场精彩的战斗，我们希望当Bob回来的时候，可以以2倍速观看错过的内容，直到追上Alice的进度后，双方再保持原始的速度进行观看。

```swift
extension AVCoordinatedPlaybackSuspension.Reason {
    static var whatHAppened = AVCoordinatedPlaybackSuspension.Reason(rawValue: "com.example.groupwatching.suspension.what-hAppened")
}

func performWhatHAppened() {
        
    // Rewind 10 seconds.
    let rewindDuration = CMTime(value: 10, timescale: 1)
    let rewindTime = player.currentTime() - rewindDuration
        
    // Start a custom suspension.
    let suspension = player.playbackCoordinator.beginSuspension(for: .whatHAppened)
    player.seek(to: rewindTime)
    player.rate = 2.0
        
    DispatchQueue.main.asyncAfter(deadline: .now() + rewindDuration.seconds) {
        // End the suspension and resume playback with the group.
        suspension.end()
    }
}
```

我们创建一个suspension对象，创建时需要输入一个AVCoordinatedPlaybackSuspensionReason，扩展并实现这个属性。在创建完成后，所有的指令就不会被`AVPlayerPlaybackCoordinator `拦截，当想要结束的时候，直接调用`end()`指令即可。并且在结束时不需要将状态重置，因为重新加入到session的时候会以session中的状态为准。

结束的另一个指令`end(proposingNewTime newTime: CMTime)`会在结束时会所有用户更新播放的时间。这也是seek时自动生成的suspension起效的方法。


### 最后的介绍：`AVDelegatingPlaybackCoordinator `
可惜的是，许多程序采用了自研的播放器，而非直接使用AVPlayer。在这种情况下，就需要使用`AVDelegatingPlaybackCoordinator `来进行状态管理。

[![RiUQ0K.png](https://z3.ax1x.com/2021/06/20/RiUQ0K.png)](https://imgtu.com/i/RiUQ0K)

`AVDelegatingPlaybackCoordinator `的使用和上文所提到的基本一致，主要的区别是，所有的播控操作需要额外的包一层。一些细节需要注意的是：
- 需要在手动实现一个delegate协议，完成seek、play、pause、buffer等操作
- UI首先通知`AVDelegatingPlaybackCoordinator `，而不是播放器。`AVDelegatingPlaybackCoordinator `来决定先和别的Coordinator沟通，还是通知播放器改变状态
- 播放器的回调也要给到`AVDelegatingPlaybackCoordinator `，所有的UI刷新也要通知给另外一端的`AVDelegatingPlaybackCoordinator `
- `AVDelegatingPlaybackCoordinator `不提供自动的suspension，需要开发者手动实现
- `AVDelegatingPlaybackCoordinator `可以和`AVPlayerPlaybackCoordinator `通讯，但是必须使用自定的item identifier
