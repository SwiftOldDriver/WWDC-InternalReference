## 二、本届 `XCTest` 值得期待的特性
我们不急着看新 API，先来看个故事。假定公司有员工 A 和员工 B 共同开发完成一个复杂模块并开始写测试用例自测。我们用代码来描述以上场景，如下：

```
class ComplexModule {
    static func stuffA_DoSomethingReturnTrue() -> Bool {
        return true
    }
    static func stuffB_DoSomethingReturnTrue() -> Bool {
        return true
    }
}

class DiagnoseWithTestRepetitionsTests: XCTestCase {
    
    func testXCTExpectFailure() throws {
        XCTAssertTrue(ComplexModule.stuffA_DoSomethingReturnTrue())
        XCTAssertTrue(ComplexModule.stuffB_DoSomethingReturnTrue())
    }
    // ...
}

```

这时，员工 A 发现代码逻辑有严重缺陷需要部分重构，员工 B 继续开发测试用例自测。假定二人在同个分支开发（别抬杠，简化问题）。那问题来了，由于 A 的重构导致 `stuffA_DoSomethingReturnTrue` 等大量功能的返回值出错，间接导致员工 B 的测试用例报告出现很多 `fail` 干扰项，严重影响自测效率。

![干扰项](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625669656241-11ab030f-d202-45fd-ab95-1d2a9b2b597c.png?x-oss-process=image%2Fresize%2Cw_748)

员工 B 有什么方式可以临时屏蔽干扰项？有 2 种方式：
1. 注释大法，把所有员工 A 的 `XCTAssert` 相关方法注释掉，并且打上标记，过段时间恢复
2. 用`XCTSkip` 相关函数跳过（相关介绍[点击这里](https://developer.apple.com/documentation/xctest/3521325-xctskipif)）：

![XCTSkip](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625671582311-1fb6612a-0ae9-44ba-8467-16969dc4b8fe.png?x-oss-process=image%2Fresize%2Cw_748)

注意到左上角，发现 `testXCTExpectFailure` 这个 test case 不再标红，`DiagnoseWithTestRepetitionsTests` 这个 Suite 也已经通过。但使用 `XCTSkip` 代价很大，我们调整了函数顺序，因为 `XCTSkip` 函数会把作用域内当前行之后的代码全部屏蔽。因此，用来屏蔽干扰并不合适。

接下来，我们一起看看更优雅的方式。

### 2.1 `XCTExpectFailure`
`XCTExpectFailure` 是 Apple 设计用来声明即将出现预期 `Assert fail` 的 API。在看完 Session 视频以及接口描述后，我个人认为非常适合在重构代码或者功能开发过程中起到代码临时屏蔽的作用，让测试的 `workflow` 更顺畅。从 Apple 的 API 注释里也可以感受到 Apple 也有相近的设计初衷：

```
@function XCTExpectFailure()
This can be used to both document and suppress a known issue when immediate resolution is not possible.
```

现在我们看下用 `XCTExpectFailure` 如何优雅解决员工 B 的困扰：

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625672813432-495874b0-c680-415a-ac54-003c981e859e.png)

可以看出，虽然员工 A 提供的 `stuffA_DoSomethingReturnTrue` 服务依旧异常（本来为 `true` 却返回了 `false`），但并不影响员工 B 的测试报告。灰色打叉代表用例内虽然有错误，但在预期内，不算作 `fail`。整个 test suite 依旧是绿色标志。

当员工 A 调整代码让 `stuffA_DoSomethingReturnTrue` 返回值恢复正常后，我们看下会发生什么：

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625761279223-b219416c-fba3-48b6-bcb0-3e61d4cf91f6.png?x-oss-process=image%2Fresize%2Cw_748)

出现了 fail ！因为系统抛出了 `UnmatchedExpectedFailure`，因为代码预期会有个失败，但实际逻辑并未抛出。**我们正好可以用依据报错的提示，逐个去掉 `XCTExpectFailure` 临时代码，优雅地完成代码重构**。


另外，接口还允许追加参数：

```
XCTExpectFailure("PRD地址是 <www.meitu.com>", options: XCTExpectedFailure.Options()) {
    XCTAssertTrue(ComplexModule.stuffA_DoSomethingReturnTrue())
}
```

第一个参数是 reason，可以在这里添加注释以及链接。比如可以插入产品文档地址或者 Jira bug 地址。如果 reason 字段带有链接，则在测试报告列表中会多出一个「昆虫」图标，点击图标会自动唤起 Mac 浏览器并打开网页。这方便了我们查看用例报告时了解更多上下文信息。挺实用的小 Tips ！

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625674088617-236b6e99-2001-4e9d-9f72-0affb9849f0e.png?x-oss-process=image%2Fresize%2Cw_748)

`XCTExpectFailure` 还有一些使用上的注意事项，下面逐条列出。

> 我习惯先解释原理，只要原理掌握了，剩下的就是 API 语法细节了，小问题~

1. `XCTExpectFailure` 也可以不带 block 参数使用，这时表示你期望作用域内 `XCTExpectFailure`之后的代码有 fail。
2. `XCTExpectFailure` 可以嵌套调用，也就是在其 block 内嵌套调用
3. 支持通过 `XCTExpectedFailure.Options` 参数灵活配置 `isEnable` 、 `isStrict`，并通过 `issueMatcher` 闭包添加触发的过滤条件。
4. reason 字段如果是文字 + 网址的格式，则网址必须用**尖括号**包起来，比如” please refer to \<www.google.com> ”，否则 Xcode 将无法识别地址。

### 2.2 `Test Repetitions`

在产品开发过程中，令程序员与测试心惊胆战的一类问题就是「非必现 BUG 」。这类问题往往由线程安全、子线程执行 UI、全局变量改变等因素导致。测试组常常通过 [Appium](https://github.com/appium/ios-uicatalog) 等框架做自动化测试，对开发而言，我们往往需要在单元测试中搭配 [Address Sanitizer](https://developer.apple.com/documentation/xcode/diagnosing-memory-thread-and-crash-issues-early)、`Thread Sanitizer` 、`Main Thread Checker` 、`Undefined Behavior Sanitizer` 等调试选项，甚至执行多遍代码才能高概率复现此类「非必现 BUG」。

在 `Xcode 13` 之前，要执行多次代码往往需要程序员编写遍历逻辑以及停止条件来实现。 `Xcode 13` 将释放开发的双手，在功能上做多次 Test 的支持。`Xcode 13` 提供了三种重复模式：

> 触发 Repetition Test 的入口可以在 testXXX 函数的小钻石图标上右键 -> 选择 「Run "testXXX()" Repeatedly...」

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625834349431-a9f45797-3b85-40e7-9188-8463224c3e45.png)

模式 1. **Stop after failure**

该模式下，测试用例会反复执行直到出现第一次 Assert Failed 或者循环次数达到 Maximum Repetitions 设定的值。

例如下面的代码：

```
    func testRepetitions() throws {
        let random = arc4random() % 50
        XCTAssertNotEqual(random, 25)
    }
```

使用 Stop after failure 模式，最大循环次数设置为 10 次：

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625841935157-7a68bb07-27e4-47c5-92e6-cd4817856228.png)

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625841732160-38ae5f26-f12e-40e9-8214-100a979054b2.png)

可以看出，在第 5 次循环时随机数刚好等于 25，导致 `XCTAssert` 断言失败，Repetitions Test 终止。

> Tips： 在 Options 中选中 「Pause on Failure」，可以在 Repetition 停止时自动断点，有助于我们做单步调试排查问题。

这个模式非常适合用于「正常情况下正确但概率性异常」的逻辑。例如，我们可以编写测试用例用于压测美图宜肤肤质检测算法鲁棒性：

```
func testMeituEveSkinTypeAlgorithm() throws {
    let oilSkinImage = SkinImageFactory.makeSkinImage(skinType: .oil)
    var skinType = MeituEveSkinAnalyzer.analyzeSkinType(with: oilSkinImage)
    XCTAssertEqual(skinType, SkinType.oil)
    
    let drySkinImage = SkinImageFactory.makeSkinImage(skinType: .dry)
    skinType = MeituEveSkinAnalyzer.analyzeSkinType(with: drySkinImage)
    XCTAssertEqual(skinType, SkinType.dry)
}
```

将重复次数设置为一个较大数值，勾选 「Pause on Failure」，下班前开始 Run ，真香~

模式 2. **Stop after success**

类似地，该模式下，测试用例会被反复执行直到出现第一次 Assert Success 或者循环次数达到 Maximum Repetitions 设定的值。功能跟模式 1 相近，只是语义相反，这里不再赘述。

模式 3. **Maximum Repetitions**

测试用例将被重复执行指定次数。

另外，Test Repetitions 还可以在 Xcode 11中 引入的 [Test Plan](https://www.raywenderlich.com/10212963-xcode-test-plans-for-ios-getting-started) 中设置，搭配 WWDC21 中的重大更新 [Xcode Cloud](https://developer.apple.com/videos/play/wwdc2021/10268/) ，可以在 Apple CI 上做循环压测，好处是不占用本地电脑，并且可以在多种机型上压测:

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625844856386-1a09d118-a0dd-4a08-b56a-a03ff74940b0.png?x-oss-process=image%2Fresize%2Cw_1496)

## 三、本届 `UITest` 值得期待的特性

WWDC21 为 iPadOS pointer（指针）、watchOS Digital Crown（数字表冠）、macOS trackpad（触控板）的 UITest 新增一些 API 用于加强 UI 自动化测试。

### 3.1 iPadOS pointer

从iPadOS 13.4开始，Apple 开始支持鼠标以及 trackpad 配件操作 iPad 。当接入这些外设后，在 iPad 界面上就会出现指针（pointer），用户可以像操作 macOS 一样操作 iPad，比如很香的双指滚动、多指点击等。

虽然 iPad 支持了 pointer 交互，但 Apple 对 pointer 的 UI 测试支持是很弱的。在 iOS 15 前只支持模拟手势、键盘等传统操作，比如 `XCTUIElement` 的拓展 `XCUIElementTouchEvents` 中提供了 `tap`、`doubleTap`、`press`、`swipe` 等手势支持。从 **iOS15** 起新增了一系列 API 用于控制 iPadOS 的 pointer 交互，让代码也能模拟鼠标、trackpad 的多种复杂操作。这些操作包在专门的 `XCUIElementMouseEvents` extension中：

```
@interface XCUIElement (XCUIElementMouseEvents)
/// 鼠标悬停
- (void)hover API_AVAILABLE(ios(15.0), macCatalyst(13.0));
/// 鼠标左键单击
- (void)click API_AVAILABLE(ios(15.0), macCatalyst(13.0));
/// 鼠标左键双击
- (void)doubleClick API_AVAILABLE(ios(15.0), macCatalyst(13.0));
/// 鼠标右键单击
- (void)rightClick API_AVAILABLE(ios(15.0), macCatalyst(13.0));
/// 鼠标长按拖拽
- (void)clickForDuration:(NSTimeInterval)duration thenDragToElement:(XCUIElement *)otherElement API_AVAILABLE(ios(15.0), macCatalyst(13.0));
/// 鼠标长按拖拽，带速率控制
- (void)clickForDuration:(NSTimeInterval)duration thenDragToElement:(XCUIElement *)otherElement withVelocity:(XCUIGestureVelocity)velocity thenHoldForDuration:(NSTimeInterval)holdDuration API_AVAILABLE(ios(15.0), macCatalyst(13.0));
/// 鼠标滚动
- (void)scrollByDeltaX:(CGFloat)deltaX deltaY:(CGFloat)deltaY API_AVAILABLE(ios(15.0), macCatalyst(13.0));
@end
```

> 注：在UITest 中所有可操作元素都是 `XCUIElement` 类型，开发者可以通过 `XCUIElement` 提供的各种模拟交互接口完成复杂的 UITest 交互操作。关于 `XCUIElement` 可以[点击这里](https://developer.apple.com/documentation/xctest/xcuielement)了解。

调用以上 API 有两个注意事项:
1. iOS 15.0+，macOS 13.0+
2. iPadOS only！

因此在编程时需要提前做版本与平台的过滤限制，否则会遇到编译失败以及「设备不支持」的运行时异常。设备限制可以使用 Apple 本次新增的 `-[XCUIDevice supportsPointerInteraction]` 方法，配合上文提到过的 `XCTSkip` API 实现。下面我们用官方的 [demo](https://developer.apple.com/videos/play/wwdc2021/10208/?time=265) 来举例：

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625887781427-fd40a4df-f687-44ea-b9a6-4d54b65b876b.png?x-oss-process=image%2Fresize%2Cw_1240)

### 3.2 watchOS Digital Crown

> Digital Crown，译为数字表冠，指的是 Apple watch 右侧的旋钮。

从 Xcode 12.5 起，Apple 开始支持 watchOS UITest，主要是常规的界面点击、Crown 的点击事件。从 Xcode 13.0 开始，新增了对 Crown 的旋转及旋转速率控制支持：

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625888790157-23f3b1e6-ae46-42c6-a012-a0c2a47eefe6.png?x-oss-process=image%2Fresize%2Cw_1496)

接口比较简单，我们依旧用官方 demo 来演示 API 的调用。要注意，接口的第 1 个参数 `delta` 是个 `CGFloat` 类型，1 表示从上往下滚动 Crown 360 度，-2 表示反向滚动 720 度。

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625888959653-13ee2feb-735a-44c9-a113-612294b2ef76.png?x-oss-process=image%2Fresize%2Cw_1192)


### 3.3 trackpad

在 iOS 13.0 之前，Apple 提供了用于滚动指定像素点的接口：

![](https://cdn.nlark.com/yuque/0/2021/png/21800174/1625909247560-885805b5-17da-4859-934e-fbfd85dbda03.png)

开发者如果要实现持续的、带方向的滑动往往做二度封装。从 iOS 15 起，Apple 为 `XCUIElement` 提供了一些便捷方法：

```
- (void)swipeUpWithVelocity:(XCUIGestureVelocity)velocity;
- (void)swipeDownWithVelocity:(XCUIGestureVelocity)velocity;
- (void)swipeLeftWithVelocity:(XCUIGestureVelocity)velocity;
- (void)swipeRightWithVelocity:(XCUIGestureVelocity)velocity;
```

其中 `velocity` 预定义了三种模式： `Default`、`Slow`、`Fast`。如果这三种模式无法满足你的要求，你完全可以基于 `scroll(byDeltaX:deltyY)` 的接口自定义一套新模式出来。
