WWDC21 推出了一个邮件扩展的新框架 MailKit。这套框架的主要作用就是提升邮件使用的体验，框架实现的功能主要通过以下四个方面体现

1. Compose 撰写邮箱
2. Action 邮件操作
3. Content blocking 内容屏蔽
4. Message security 内容加密

四个功能分别对应着四个不同的协议：`MEComposeSessionHandler` `MEMessageActionHandler` `MEContentBlocker` `MEMessageSecurityHandler`

## Compose

Compose 扩展的功能都定义在 MEComposeSessionHandler 这个协议里。在开发 Compose 扩展的时候，需要在 Info.plist 文件中，`MEComposeSession` 字典里，指定一个图标和工具的描述性文字。撰写邮件窗口内就会显示这个按钮，包含一个图标和工具的描述性文字。
![](https://images.xiaozhuanlan.com/photo/2021/6c0bd6779769f61ac8395d4747ded66f.png)

在 `MEComposeSessionHandler` 协议里的方法，都包含了 `MEComposeSession` 这个参数类型。
邮件为每个邮件撰写窗口都创建了一个唯一的 `MEComposeSession` 实例。它有一个 `MEMessage` 属性，用于公开正在编辑的消息的各种详细信息。这些信息会在调用扩展程序时做对应的处理，不同的处理规则可以通过视图控制器进行选择和调整。
自定义消息表头 `headers` 这里先跳过，在后续中 `action` 再提到。
下面是方法的一些简单描述

![](https://images.xiaozhuanlan.com/photo/2021/23f5c5cfa53aa968d76b187ad1492916.png)

### 验证收件人邮箱地址

每次在输入地址之后，会调用扩展中验证邮箱地址的有效性的方法，然后给出对应的状态。状态里分为 success/error/wraning 三种情况

![](https://images.xiaozhuanlan.com/photo/2021/f5f16e97fe7b0b3c222d6f847d9c8b47.png)

### 视图控制器

这里的控制器，需要继承自 ` MEExtensionViewController ` 。通过视图控制器，我们就能够自己选择需要的上下文，然后来帮助我们提升在撰写邮件的使用体验。

![](https://images.xiaozhuanlan.com/photo/2021/ae82190324c214de6a746077a414c14b.png)

### 邮件信息验证

在发送邮件之前，还可以对整个邮件内容做一次验证。通过 `allowMessageSendForSession` 方法可以抛出信息验证的错误。包括邮件里包括的发送接收地址，还有邮件的内容。要注意的是邮件的正文内容是以 Data 的形式返回，而其他信息都是 `String` 的形式。

## Action 

action 主要用来处理新接收的邮件。这里需要注意：只是处理新接受的邮件，原有的邮件是不会受到影响的。处理的方式包括：

1. 标记邮件列表中显示的背景色
2. 移动邮件到对应的邮件文件夹
3. 修改邮件的读取状态和标记

![](https://images.xiaozhuanlan.com/photo/2021/7133710ba14c6408610b43fb842f1880.png)

在 session 里，实现的是将邮件列表的背景色修改为红色。
这里的 `message.headers` 就是在 compose 扩展方法 `additionalHeaders:` 里拿到的。

![](https://images.xiaozhuanlan.com/photo/2021/bfa5ebb9a2c5420bb34192fa71063f27.png)

第一次调用 `decisionAction for message` 方法时，邮件其实在列表中还未可见。MEMessage 实例将只有消 息头的一个子集，也就是只包含了 headers 数据。如果需要完整的消息正文和标题才能返回对应的 action 操作，那么可以直接返回 invokeAgainWithBody。等内容下载完成之后再重新调用一次方法。

## Message Security

信息的安全扩展功能，提供了一种更安全的邮件传输方式

1. 消息编码和解码
2. 消息签名
3. 查看签名信息证书

MEMessageSecurityHandler 协议继承自 `MEMessageDecoder` 和 `MEMessageEncoder` 

![](https://images.xiaozhuanlan.com/photo/2021/70eae7182d096c2a4e699e490772d4ca.png)

### MEMessageDecoder

对消息编码分为两部分。第一部分是获取是否可以编码的状态，方法调用发生在撰写邮件的时候

通过查询发件人和收件人列表，扩展程序会调用 `getEncodingStatus` 方法来验证是否可以签名和加密。这时候会突出显示锁定和证书图标。在发件人和收件人发生改动的时候，也会调用方法重新验证

![](https://images.xiaozhuanlan.com/photo/2021/c9ef7373586ce582a23cd55931fa6c81.png)

第二部分实际加密和签名，是在消息发送的时候实现

消息发送时，会将邮件的消息数据进行签名和加密得到结果，然后再发送出去：

![](https://images.xiaozhuanlan.com/photo/2021/2e2781da00c17233d757d342f40bad9c.png)

官方示例代码里底层的实现方式如下图。其中 `isSigned` 和 `isEncrypted` 参数应该由 ecode 方法传递进来的：

### MEMessageEncoder

得到加密的结果，如果无法解密，那么就直接返回 nil

![](https://images.xiaozhuanlan.com/photo/2021/7785aa5a0486a7ba017da8af0d4bc64a.png)

在接收到加密和签名邮件时会在列表中展示对应的图标，实现 `extensionViewController` 方法就可以获得签名信息 `MEMessageSigner`，通过视图展示完整的签名信息。

## 结语
因为是一个全新的框架，所以本文主要以扫盲为主。另外 Mac 原生邮件软件总体来说用户不是很多，而且就连 Safari 的扩展生态都不是很好，就更别谈邮件的扩展了。不过从扩展的思路来说，一个扩展能在多个 App 里去使用，这个想象的空间就大了。
不过从开发上来说也是学习官方代码的一个好机会。功能不复杂，结构也很清晰。可以学习以协议的实现逻辑来实现现有功能的扩展需求。
