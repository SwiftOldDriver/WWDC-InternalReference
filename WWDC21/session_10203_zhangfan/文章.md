## 前言

使用 Xcode 开发过的同学，对 Organizer 一定不陌生。在 Organizer 里可以查看 AppStore 和 TestFlight 收集的崩溃日志，以及设备使用期间的耗电、性能、以及磁盘读写等数据。 [WWDC21](https://developer.apple.com/wwdc21/) 中，Xcode 团队对 Organizer 中部分功能进行了升级，让我们一起来看看今年更新的内容吧。😃

本期 Session 主要内容：

1. Organizer 中 Crashes 模块新增 6 项特性，提高开发者获取崩溃信息效率，减少修复崩溃时沟通成本。
2. Metrics 中增加 Terminations 性能指标，汇总展示 app 被系统关闭的日均次数，与不同 app 版本之间的分布图。

在梳理 Crashes 模块新特性之前，笔者想介绍 4 个崩溃相关的知识点：

### 什么是崩溃？

崩溃是指当程序执行一些不被允许的行为时，发生的异常退出现象。

### 崩溃是怎么产生的？

- 程序会在执行无意义计算时发生崩溃，比如除以 0。

- 应用会在启动阶段耗时超过 20s 时发生崩溃，因为在 Apple 的 [看门狗](https://developer.apple.com/documentation/xcode/addressing-watchdog-terminations) 机制下，启动阶段耗时过长，应用会被系统强制关闭。并在崩溃日志中标记本次崩溃原因：`0x8badf00d`（发音 ate bad food，表示吃到了坏东西），启动超时甚至还会使应用审核时被审核人员拒绝。

  崩溃日志中的部分信息：

  ```
  Exception Type:  EXC_CRASH (SIGKILL)
  Exception Codes: 0x0000000000000000, 0x0000000000000000
  Exception Note:  EXC_CORPSE_NOTIFY
  Termination Reason: Namespace SPRINGBOARD, Code 0x8badf00d
  ```

- 应用使用的内存超过了系统限制时，就会发生 [OOM（Out-Of-Memory 超出内存限制）](https://en.wikipedia.org/wiki/Out_of_memory)，触发 [Jetsam](https://developer.apple.com/documentation/xcode/identifying-high-memory-use-with-jetsam-event-reports) 机制强制关闭应用。Jetsam 机制是 iOS 操作系统中，为了防止内存资源滥用而衍生出的一套机制，当应用过度占用内存空间时，会被系统强制关闭，并生成一份格式为 JSON，前缀为 JetsamEvent 的文件保存在手机本地。由于内存不足而引起的系统强杀，大多数情况无法记录堆栈信息，大多数第三方也只能简单的采集次数。实际想要调试的话，还是得靠真机导出报告进行分析。

- 当程序访问数组越界，以及错误操作野指针等等，都会产生崩溃。

### 如何获取崩溃日志？

开发可以执行测试发现程序中隐藏的问题，还可以根据测试同学提供的复现路径定位崩溃原因，解决应用发版前发生的崩溃。当崩溃出现在线上时，处理起来就变得十分棘手。

如何高效的修复线上的问题呢？应用的线上崩溃日志可以有效还原崩溃发生时的堆栈上下文，我们可以利用日志解决大部分的线上崩溃。

下面介绍常见的获取崩溃日志方法：

- 连接真机设备导出对应的崩溃日志：Xcode -> Window -> Devices and Simulators -> View Device Logs。
- 从运行设备中导出崩溃日志：Settings -> Privacy -> Diagnostics & Usage -> Diagnostic & Usage Data。
- 运行 Xcode，下载用户 [共享 ](https://support.apple.com/zh-cn/HT202100)的崩溃日志：Xcode -> Window -> Organizer -> Crashes
- 引入第三方的统计工具，如 Bugly、Firebase、App Center 等。

### 了解崩溃日志基本格式

发生崩溃时，系统自动保存本次的崩溃日志到设备中，崩溃日志主要由以下信息组成：

（以下的崩溃信息来自真机）

1. 基本环境信息 - 崩溃发生时间、应用进程号、版本号以及运行设备的硬件信息等等。

   ```

   Hardware Model:      iPhone11,8
   Process:             Video [1451]
   Version:             1 (4.0.0)
   Code Type:           ARM-64 (Native)
   Coalition:           com.video.app [470]
   
   Date/Time:           2021-06-17 16:49:02.6683 +0800
   Launch Time:         2021-06-17 16:48:59.3579 +0800
   OS Version:          iPhone OS 14.3 (18C66)
   ```

2. 崩溃原因信息 - 崩溃类型、错误码、异常标记、终止原因与描述。

   ```

   Exception Note:  EXC_CORPSE_NOTIFY
   ```

3. 堆栈信息 - 保留了程序终止时，每个线程的堆栈信息。

了解崩溃的产生，崩溃日志基本结构、日志收集与上传以及相关流程，能够提高解决崩溃的效率，与保障应用的稳定性。如果想知道更加详细的崩溃日志与治理细节，安利一份 Session [搞懂崩溃与崩溃日志](https://developer.apple.com/videos/play/wwdc2018/414/)。视频中，详细介绍了崩溃与崩溃日志，并且结合实际崩溃案例演示在遇到崩溃时，我们应该如何分析与解决报错。

复习完什么是崩溃与如何找到崩溃日志后，接下来以介绍今天的重点：崩溃日志模块。

## Crashes 目前功能

1. 展示从 App Store 与 TestFlight 中收集到的崩溃日志，按出现次数排序

小总结

在本次更新中，Xcode 团队重点加强了日志获取效率与筛选能力，帮助我们更高效地获取 TestFlight 当前正在发生的崩溃。美中不足的是，目前崩溃模块仍然不支持更加自定义的时间范围筛选，比如查询本月指定几天发生的 TestFlight 崩溃日志，在进行负盘工作时，只能通过收集日志到本地再进行时间块筛选。虽然目前的日志列表可以即时显示最新的崩溃信息，但是在界面上没有设置主动刷新的按钮，只能等待被动的更新，期待能在未来的更新中看到刷新按钮。

## 新增 Terminations

从最近几年以来的 WWDC 相关 Session 数据可以观察到，几乎每一年，Apple 都会在 Xcode Organizer 中添加帮助 app 性能优化的相关组件。不难看出，Apple 非常重视用户在使用应用过程中的体验，不断迭代 Xcode 来帮助全球开发者优化应用的性能表现。

WWDC21，Xcode 13 中新增 Terminations 模块，展示应用被系统强制关闭的日均数据。

![terminations_feature](https://gitee.com/alex811/wwdc_10203/raw/master/terminations_feature.png)

**Terminations 特性**

- 展示应用被系统强制关闭的日均数据
- 包含按照关闭原因分块的柱状图
- 区分不同应用版本
- 展示应用在前/后台被关闭的数据

介绍 Terminations（强制关闭统计）模块，不得不提的就是为什么我们的应用会被强制关闭？甚至有时候它运行好好的，连崩溃都没有，就被系统强行终止了。

有许多原因都会导致系统强制关闭当前应用，有的来自用户的上滑退出，也有的是因为程序运行过程发生了崩溃，也有可能在后台期间使用的内存过大，导致系统对 app 进行强制关闭。其中最容易引起系统强制清理机制就是错误地使用内存：

1. 当内存使用超过限制时，会被系统清理。每种机型都有内存使用限制，比如 iPhone 6s 之前机型，内存峰值达到 200m 以上的话，就有可能触发 Jetsam 被系统关闭。

2. 切换到后台中，如果应用没有释放一些不必要的内存，也有可能被系统清理。也就是当程序没有发生崩溃时，也会被系统清理。这是因为系统为了给其他应用腾出可用空间，而杀掉后台的应用程序的性能保护行为。


那么我们该怎么做，才能尽可能地不被系统清理？

进入后台前，可以尝试保存当前的应用状态，释放在后台时无用的内存，最好可以保持在 50MB 以下。可以有效地减少被自动清理的概率。在恢复进入应用后，读取缓存状态进行恢复。这样在后台保活的前提下还能保证用户操作的连贯性。

## MetricsKit 新特性

MetricsKit 能够收集设备耗电量以及各项性能指标。通过 MetricsKit 能够记录代码执行时，应用的各项性能数据。（所有收集到的性能数据，都是匿名的，不用担心隐私的风险）

**WWDC21 之前：**

![metrickit_playloads](https://gitee.com/alex811/wwdc_10203/raw/master/metrickit_playloads.png)

MetricsKit 会在一天结束后，汇总过去24小时的性能数据，然后在下一次启动后，通过代理回调数据。

摘自 [官方文档 ](https://developer.apple.com/documentation/metrickit)的一句话：

> With MetricKit, you can receive on-device app diagnostics and power and performance metrics captured by the system. A registered app receives reports containing data about the previous 24 hours at most once per day.

也就是说，每天最多只会收到一次过去 24 小时的统计报告。

**WWDC21 之后：**

1. 应用每次重新启动后，都会上报 [MXCrashDiagnostic](https://developer.apple.com/documentation/metrickit/mxcrashdiagnostic) 设备崩溃诊断报告。
2. MetricKit 开始支持 macOS（macOS 12.0+）

在本期 Session 中，主要概括了 MetricsKit 一项新特性以及新增对 macOS 的支持。想要深入了解 MetricKit 的同学，可以查看 [又见 MetricKit ](https://developer.apple.com/videos/play/wwdc2020/10081/) 中的分享内容，在该话题中，详细介绍了性能指标类型，以及每个指标对应的含义，并包含了实际的代码演示，帮助开发者快速入门 MetricKit 。

在对设备性能的提高有了了解与思考之后，可以继续往下阅读 [合理地使用 Xcode Organizer 来提高应用的性能](https://developer.apple.com/wwdc20/10076) 与 [App 终极性能优化指南](https://developer.apple.com/videos/play/wwdc2021/10181) 这两份指南。从工具的使用维度来演示 app 性能分析与优化。尤其是在后者的优化指南中，向我们介绍了耗电量、页面渲染等各项性能指标在 Xcode Organizer 中是如何被观察与使用的，相信在看完指南后，会对 MetricsKit 有更进一步的了解。

## 结语

今年的 WWDC21，Xcode 团队增强了崩溃日志 Organizer 的筛选功能，协助开发者在处理 App Store/TestFlight 崩溃时，更快速地获取到最新的日志列表。特别是崩溃日志的分享功能，不论是对企业团队，还是独立开发者，都能带来非常直观的收益。结合今年 WWDC21 大会上公布的 [Xcode Cloud](https://developer.apple.com/videos/play/wwdc2021/10267/) 来看，Apple 团队也注意到持续集成与持续交付已经成为开发闭环中，不可或缺的环节。对开发者而言，提高产品质量是我们的核心目标之一，希望在明年的 WWDC 中能看到更多提升质量的解决方案。
