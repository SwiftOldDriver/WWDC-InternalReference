# WWDC2021-10202 Detect bug early using static analyzer

这篇文章向你展示如何在 Xcode 上使用 LLVM 内置的静态分析器（static analyzer）帮助你找到Bug并且修复它。

所以我们的主题就是**静态分析器**。什么是静态分析器呢？

>Static analysis is the analysis of programs that is performed without actually executing the programs.

### 尽早地发现Bug

静态分析器是一个 Xcode 内置的工具，使用它可以在不运行源代码的状态下进行静态的代码分析，并且找出其中的Bug，为app提供质量保证。

另外它可以帮你找到代码中没有被测试用例覆盖到的代码执行路劲。

前面提到它是一个 Clang 体系下的工具，所以静态分析器只支持 C/C++/Objective-C 语言，当然，Objective-C 和 Swift 汇编的工程也可以很好地支持。

下面是一个实际的使用案例，这是一个正在开发中的 APP —— Solar System，使用 Objective-C 和 Swift 汇编的展示太阳系视图的 app。然后通过点击 Xcode 的 Product 菜单栏中的 Analyze 选项就可以让分析器开始工作：

像编译一样，需要等待一段时间，然后被静态分析器捕获的问题就以蓝色的感叹号 在Xcode 上方显示出来

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237769293296.jpg?x-oss-process=image/auto-orient,1/quality,q_90)


也可以在 issue navigator 中查看蓝色的分析器捕获的问题

以下是目前分析器支持捕获的错误类型，更详细的请查看本篇文章的「常见捕获汇总」篇章

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237769394299.jpg?x-oss-process=image/auto-orient,1/quality,q_90)


其中包括了安全、逻辑、以及API方面的问题

分析器可以帮你找到以上的这些问题，并且解释原因以及指出代码的执行路劲。

这里是 Solar 代码中被分析器捕获的一个问题：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237769612463.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

可以看到这个方法声明了返回类型为 nonnull，但是分析器发现了可能返回为 nil 的情况，这并不符合返回值定义的预期。如果想要查看返回为 nil 的具体原因，我们可以通过点击左边 issue navigator 中的这个 issue，点击展开后的具体步骤，查看分析器分析到的导致问题产生的代码执行路径。

点击导致返回 nil 的最后一个步骤，可以看到：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237769806118.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

自底向上查看，最后一个步骤是 `position` 这个变量初始化为 nil。通过点击倒数第二个步骤可以看到，`regularPositionAtDate` 这个方法没有被调用，因为方法的接收者是 nil。而我们知道，在 OC 中调用方法（也叫消息发送）的时候，如果方法的调用者（也被称为：消息的接收者）本身为 nil，方法调用的结果会返回 nil，所以这行代码的执行结果就是 `position` 依然为 nil。

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237769878541.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

那么为什么方法的接收者 `object` 变量是 nil 呢？继续向上查看我们发现是因为这个 switch 语句中的 default 分支没有做任何的处理，这样 object 对象就会保持初始化的状态，即为 nil。

找到了原因，为了修复这个问题，通过把第一个 case 中的处理作为 default，对 `object` 赋值为 self 即可，修复之后再次运行分析器，确保这个 bug 被成功修复。

如果这个 bug 没有被修复，在这个 OC 和 Swift 汇编的项目中，如果 Swift 调用了这个声明返回值为 nonnull 的方法，并且返回了 nil，会在运行时导致意外发生！

这样我们加借助分析器完成了一个潜在问题的修复！

### Xcode 13 新增的检查项

在 Xcode 13 中，静态分析器也得以升级，现在它可以捕获更多的一些逻辑问题：

1. 死循环
2. 无用的冗余代码（例如多余的分支条件）
3. 断言 Assert 的副作用
4. C++ 中 move 和 forward 的滥用导致的潜在问题

一部分的改进来自于开源作者们对 Apple Clang 编译器的贡献。

下面举例这些新的检查项捕获：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770012123.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

经常使用 NSAssert 是很常见的来规避非预期代码逻辑的好习惯，但是主要注意规避一些副作用，例如在NSAssert的判断条件中操作变量或内存。

上图例子中，依旧是太阳系系统的案例，数组中是一个个的天体对象，遍历天体，计算有卫星的天体数量，由于这个计数变量 `objectsWithMoonsCount` 是在NSAssert中做的累加操作，但是在Release模式中，为了程序的执行效率，会把assert给禁用，这样这个累加的操作就不会在Release模式下被执行，这样就导致了业务错误。

这样的错误在 Xcode 13 的静态分析器中可以被捕获出来，不仅仅是 Objectie-C 的 NSAssert，也包括了 C 和 C++ 的 assert 函数：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770084392.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

当然这个代码也很好修复，我们只需要把累加操作外置到NSAssert函数的上面即可：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770150928.jpg?x-oss-process=image/auto-orient,1/quality,q_90)


下面是一个很常见的死循环的案例，这种稍微复杂一些的逻辑，乍眼一看，似乎没有什么问题：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770245970.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

这段代码中，做了一个二维循环，但是在内层的循环中，没有对 `column` 做递增，而是做了 `value` 的递增，这个不是特别容易发现，但是新版本的静态检查器不会可不会错过这个问题，他会捕获这个问题，如图：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770353074.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

把 `value` 替换成 `column` 修复这个问题：

### 自定义分析器

Xcode 也为静态分析器提供了很多的自定义项，方便开发者集成到开发工作流当中，我们可以通过BuildSetting中的设置，使得每一次编译都去运行静态分析器。

##### 每一次编译都执行分析器

在BuildSetting中通过搜索 `analysis` 关键字，可以筛选出跟分析器相关的设置项，通过打开 `Analyze During 'Build'` 可以使得每一次编译操作都执行分析器：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770540286.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

##### 分析器运行模式

另一个选项，`Mode of Analysis for 'Analyze'` 可以配置分析器运行的模式，Xcode 提供了两种运行模式：`Shallow(faster)` 和 `Deep`，顾名思义，`Shallow`规避了去检查一些耗时复杂的检查操作，所以 Shallow 运行的更快，`Deep` 则进行深入的检查，能抛出更全面的错误。

##### 专项检查配置

另外也提供了一些专项检查的配置项目，可以根据工程情况定制选择：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770661442.jpg?x-oss-process=image/auto-orient,1/quality,q_90)


例如，如果项目有严格的安全检查，可以打开下图中选中的这些配置项目：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770718180.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

如果你发现一些检查项目跟你的代码无关，你想忽略掉它，可以在下面这些选项中逐个地关闭特殊的一类，从而使得抛出的捕获仅是你想要关注的问题：

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770780715.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

##### 单个文件的分析

我们也可以对单个文件进行分析，选中你想要分析的文件，在 `Product` 菜单栏下选择 `Analyze ’FileName‘` 即可进行对这个文件进行分析，并且不会分析import关联的所有文件。

![](http://yangzai360.oss-cn-beijing.aliyuncs.com/2021/06/16/16237770866586.jpg?x-oss-process=image/auto-orient,1/quality,q_90)

这个操作非常方便，不需要编译就可以对我们修改的代码进行分析。
