数据竞争在多线程编程中是一个让人头疼的问题，造成这种问题的原因主要是同一个资源被多个线程访问，并且有读写操作。那么解决这种问题的方式我们主要是用同步或者加锁，这种方式用起来都需要非常小心，否则会造成死锁。在今年WWDC大会中给我带来一种新的的避免数据竞争的方式，今天我们看一下这种新的方式-Actor。

## Actor

#### 数据竞争

在多线程操作中，多个线程访问同一个数据的时候，很容易造成数据竞争，我们先看一下下面的代码

```swift
class Counter {
    var value = 0

    func increment() -> Int {
        value = value + 1
        return value
    }
}

let counter = Counter()
asyncDeteched {
    print(counter.increment()) // 1 or 2 
}

asyncDeteched {
    print(counter.increment()) // 1 or 2 
}
```

我们可以发现在多线程中在不同的时机操作`value`,输出的结果是不一样的，有可能是`1`也有可能是`2`，这就造成了数据竞争。如何避免这种问题出现呢。我们常用的方法就是加锁，或者同步的方法来解决问题，但是我们在使用这几种方式的时候要非常的小心，否则会出现不必要的问题，比如死锁。

这个时候`Actor`就出现了，解决了多线程中的资源竞争问题。

[Actor model](https://en.wikipedia.org/wiki/Actor_model) 是一种多线程问题解决方案，由状态（state）、行为（behavior）、邮箱（mailbox）三者组成：

1. 状态（state）：状态是指actor对象的变量信息，状态由actor自身管理，只能在actor内部访问和改变。

2. 行为（behavior）：行为指定的是actor中计算逻辑，通过actor接收到的消息来改变actor的状态。

3. 邮箱（mailbox）：邮箱是actor之间的通信桥梁，邮箱内部通过FIFO消息队列来存储发送发消息，而接收方则从邮箱中获取消息

Actor中可变属性被访问的时候将程序其他的部分与之隔离起来，保证属性可以被同步访问，

- 所有的属性被访问的时候都是通过Actor
- 访问属性的时候提供互斥性

#### Actor Types

 在 Swift 中 Actor  是和类一样是引用类型，也有属性，方法，下标，遵守协议等功能。和类最大的区别在于访问内部数据的机制，Actor访问数据的时候会将数据隔离起来，每次只能一个线程访问内部数据，保证数据的操作安全，Actor 声明是用 `actor` 关键字，下面我们看一下用 `actor` 来解决上面问题。

```swift
actor Counter {
    var value = 0

    func increment() -> Int {
        value = value + 1
        return value
    }
}

let counter = Counter()
asyncDeteched {
    print(counter.increment()) // 1 or 2 
}

asyncDeteched {
    print(counter.increment()) // 2 or 1 
}
```

这样在actor的隔离中，在多线程访问只能单个线程访问，在一个线程访问的时候，另一个线程在等待，两个线程依次访问，不会造成数据的竞争，输出的结果也是 `1 2` 或者 `2 1`。

#### Actor isolate

在actor中防止数据竞争的主角就是Actor isolate，  在actor中主要的保护机制就是只有`self`可以直接访问内部存储变量，在外部访问是不被允许的。如果我们给Count 增加一个减法的操作，并操作了另一个actor 的 value变量，这样在编译器层面就报错了。

```swift
extension Counter{
  func subtract(other: Counter) throws {
    other.value = other.value - value  // error: actor-isolated property 'value' can only be referenced on 'self'
  }
}
```

在actor声明的存储变量和方法都是被`actor-isolated` 隔离的，只能被self访问。而value 是可变的存储变量 `other.value` 不是通过self访问的，在编译器层面就是不通过的，在actor 之间可以传递的智能是immutable 变量，或者let变量

### 可重入性

我们看一个图片下载的例子

```swift
actor ImageDownloader {
    private var cache: [URL: Image] = [:]

    func image(from url: URL) async throws -> Image? {
        if let cached = cache[url] {
            return cached
        }

        let imageData = try await downloadImage(from: url)
        let image = try await Image(decoding: imageData)
        // Potential bug: `cache` may have changed.
        cache[url] = image
        return image
    }
}
```

这是一个图片下载的过程，我们可以看到当 `cache` 里面不含有需要下载的图片的时候，会走到下载图片的步骤`downloadImage` 并将线程挂起，但是当一个相同的url也进行下载时，首先查缓存，由于前面图片还没有下载完毕，所以还需要走下载的步骤，并等待前面图片下载完成，再进行下载，如果当图片下载完成时服务器那边图片更改，但还是同一个url，当第二次url下载图片并进行存储时，会造成相同的url但是返回不一样的图片，这是 一个隐藏的bug。所以在我们await 操作之后，程序中的其他变量会发生变化，为了不发生这种潜在的错误，我们需要在await之后，要进行相应的判断。

这就是actor的可重入性，可以多次进入某个方法，但是这样有什么好处呢？假如，我们没有这个可重入性操作，每次进入都需要等待前面一次的结果，当前下载图片操作只有两个耗时操作，如果耗时操作再多几个效果更明显。如果没有可重入性，每次都需要等待前面整个步骤全部完成之后才能进行，效率太低了，在可重入特性下，在每个await都能有一个线程进行操作，速度会提升很多。还有一个重要的特点是可重入性可以在多个线程等待中，可以优先选择优先级高的线程进行下一步操作 。

### Sendable

Sendable意思就是在并发编程中能够安全的传递的数据就可以称为Sendable。Sendable是一个协议，遵守这个协议的都是Sendable

```swift
protocol Actor : AnyObject, Sendable { }
protocol Sendable { }
```

所有的actor都是遵守Actor协议，Actor协议遵守Sendable协议，所以所有的actor都是Sendable。根据协议我们可以看出可Sendable的主要是可同步操作的对象，或者不可变的对象，主要有以下几种：

- 值类型
- Actor 类型
- 不可变的类(属性都是let修饰的)
- 内部都是同步操作的类
- @Sendable function

值类型在传递的过程是可以独立的复制一份的，所以他是sendable的。Actor类型在 操作的时候是同步操作的，且遵守Sendable协议。

不可变的类在操作的时候是不可变的数据也是安全的。函数也可能是Sendable，闭包是也是函数，闭包也可能是Senable，必要的条件就是：不能捕获变量，或者捕获的都是Sendable变量，不能是逃逸闭包。

### MainActor

MainActor在程序中是一个全局的单例，他能保证所有的修改都能在主线程中更新

```swift
@globalActor public actor MainActor {
    public static let shared: MainActor
}
```

我们也可以将一个方法或者类声明一直只能在主线程中使用 ，用`@MainActor`

```swift
@MainActor func runInmain(){
  ...
}
```

这个方法是中运行在主线程中

```swift
@MainActor actor Count{
  ...
}
```

属性和方法调用会也只会在主线程中调用

这里你看到了如何用Actor优雅的解决线程数据竞争问题，并切在线程中用Sendable数据在多线程中安全的传递数据，避免了使用锁等其他工具的繁琐，最后MainActor保证了我们的代码是中运行在主线程，这都是Swift5.5带给我们的惊喜，期待用Actor去构建安全的高效的SwiftCode，用这些工具解决遇到难以解决的问题。
