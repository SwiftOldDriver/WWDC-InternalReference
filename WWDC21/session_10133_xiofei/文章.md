# WWDC21 10133 - 并发编程新利器 - Actor


![](https://cdn.nlark.com/yuque/0/2021/png/1319434/1624510587647-7cccb0c9-56b9-4ffd-a028-c272f7d9bde9.png)  

数据竞争在并发编程中是一个让人头疼的问题，造成这种问题的原因主要是同一个资源被多个并发任务同时进行读写操作。那么解决这种问题的方式我们主要是用同步或者加锁，这两种方式用起来都需要非常小心，否则会造成死锁。在今年 WWDC 大会中给我们带来一种新的避免数据竞争的方式，今天我们看一下这种新的方式 - Actor。

> 本文基于 WWDC21 [Session 10133](https://developer.apple.com/videos/play/wwdc2021/10133/) 梳理。  

## Actor
#### 数据竞争

在并发编程中，多个并发任务访问同一个数据的时候，很容易造成数据竞争，我们先看一下下面的代码：

```swift
class Counter {
var value = 0

func increment() -> Int {
value = value + 1
return value
}
}

let counter = Counter()
asyncDeteched {
print(counter.increment()) // 1 or 2
}

asyncDeteched {
print(counter.increment()) // 1 or 2
}
```

我们可以发现在多线程中在不同的时机操作 `value`，输出的结果有可能是`1`和`1`，也有可能是`2`和`2`，这就造成了数据竞争。如何避免这种问题出现呢？答案是 Actor，Swift5.5 里为我们带来了新的并发计算模型。

Actor 是一个和 class 类型很相似的新类型，它可以保证在并发环境中，可变状态被访问的安全性，它只在内部维护自己的可变状态，外部访问 Actor 内部数据的时候，会按顺序来处理。可能对于 iOS 开发者来说，最直白但不准确的理解方式，就是把每个 actor 都当成是一个后端服务：

1. 我们没办法直接访问它在内存里的数据（actor isolation）；

2. 只能给它发消息请求（mail），调用它规定好的接口（behavior）；

3. 这台服务器会按顺序逐一去处理这些消息请求（mailbox + FIFO），避免并发访问自己的内部数据。

这样保证了访问数据的安全性，解决了并发环境带来的问题。

#### Actor Types

在 Swift 中 Actor 是和 Class 一样是引用类型，也有属性，方法，下标，遵守协议等功能，但是在静态方法中是没有 self 的，所以不涉及到数据的隔离。

Actor 和 Class 区别在于 Actor 遵守了 Actor 协议，可以保证在并发环境下操作数据的安全性，而且 Actor 不支持继承，使得 Actor 使用起来很简单。所以 Actor 没有重写方法的能力，这在未来可能会改变。

Actor 声明是用 `actor` 关键字，下面我们看一下用 `actor` 来解决上面问题。

```swift
actor Counter {
var value = 0

func increment() -> Int {
value = value + 1
return value
}
}

let counter = Counter()
asyncDeteched {
print(counter.increment()) // 1 or 2
}

asyncDeteched {
print(counter.increment()) // 2 or 1
}
```

这样在actor的隔离中，在多个并发任务中，保证资源被访问的同步性，在一个并发任务访问的时候，另一个在等待，两个任务依次访问，不会造成数据的竞争，输出的结果也是 `1 1` 或者 `2 2`。

#### Actor isolation

在 Actor 中防止数据竞争的主角就是 Actor Isolation，在 Actor 中主要的保护机制就是只有 `self` 可以直接访问内部存储变量，在外部访问是不被允许的。如果我们给 Count 增加一个减法的操作，并操作了另一个 Actor 的 value 变量，这样在编译器层面就报错了。

```swift
extension Counter{
func subtract(other: Counter) throws {
other.value = other.value - value  // 错误: 'value' 属性只能被'self'饮用
}
}
```

在上面的代码我们看到一个错误："被隔离的属性 `value` 只能被 self 引用"，这告诉我们在 actor 声明的存储变量和方法都是被 `actor-isolated` 隔离的，只能被 self 访问，跨 actor 边界传递非 sendable 类型会报一个错误。产生错误的原因就是 value 是可变的存储变量，`other.value` 不是通过 self 访问的，在编译器层面就是不通过的，在 actor 之间可以传递的只能是 Sendable 类型。

### 可重入性

我们看一个图片下载的例子：

```swift
actor ImageDownloader {
private var cache: [URL: Image] = [:]

func image(from url: URL) async throws -> Image? {
if let cached = cache[url] {
return cached
}

let imageData = try await downloadImage(from: url)
let image = try await Image(decoding: imageData)
// 潜在的 bug: `cache` 可能会被改变。
cache[url] = image
return image
}
}
```

这是一个图片下载的过程，我们可以看到当 `cache` 里面不含有需要下载的图片的时候，会走到下载图片的步骤 `downloadImage` 并将任务挂起，但是当一个相同的 url 也进行下载时，首先查缓存，由于前面图片还没有下载完毕，所以还需要走下载的步骤，并等待前面图片下载完成，再进行下载，如果当图片下载完成时服务器那边图片更改，但还是同一个 url，当第二次 url 下载图片并进行存储时，会造成相同的 url 但是返回不一样的图片，这是一个隐藏的 bug。所以在我们 await 操作之后，程序中的其他变量会发生变化，为了不发生这种潜在的错误，我们需要在 await 之后，要进行相应的判断。

这就是 actor 的可重入性，如果有被挂起的任务，它不需要等待被挂起的任务执行完成，就可以处理下一个任务。但是这样有什么好处呢？假如，我们没有这个可重入性操作，每次进入都需要等待前面一次的结果，当前下载图片操作只有两个耗时操作，如果耗时操作再多几个效果更明显。如果没有可重入性，每次都需要等待前面整个步骤全部完成之后才能进行，效率太低了，在可重入特性下，在每个 await 都能有一个任务进行操作，速度会提升很多。还有一个重要的特点是可重入性可以在多个任务等待中，可以优先选择优先级高的任务进行下一步操作。

### Sendable

Sendable 意思就是在并发编程中能够安全的传递的数据就可以称为 Sendable。Sendable 是一个协议，遵守这个协议的都是 Sendable：

```swift
protocol Actor : AnyObject, Sendable { }
protocol Sendable { }
```

所有的 actor 都是遵守 Actor 协议，Actor 协议遵守 Sendable 协议，所以所有的 actor 都是 Sendable。根据协议我们可以看出可 Sendable 的主要是可同步操作的对象，或者不可变的对象，主要有以下几种：

- 值类型
- Actor 类型
- 不可变的类（属性都是 let 修饰的）
- 内部都是同步操作的类
- @Sendable function

值类型在传递的过程是可以独立的复制一份的，所以是 Sendable 的。Actor 类型在操作的时候是同步操作的，且遵守 Sendable 协议。

不可变的类在操作是不可变的数据的时候也是安全的。函数也可能是 Sendable，闭包是匿名函数，所以闭包也可能是 Sendable。而闭包要成为 Sendable 的必要的条件就是：不能捕获变量，或者捕获的都是 Sendable 变量，并且不能是逃逸闭包。

### MainActor

在构建程序的时候，UI 视图的构建和处理用户事件的交互我们会在主线程中完成，但是有一些耗时的操作，比如网络请求和文件的读写操作，为了不阻塞主线程需要将这些任务放到其他线程，等执行完之后再通过 `DispatchQueue.main.async {...}` 回到主线程修改 UI，但这样容易错漏，只能通过文档或者是运行时报错来提醒外部调用者。现在我们多了一种更简洁的方式，可以使用 `@MainActor` 去标注那些 UI 相关的属性和方法，编译器会保证这些属性和方法只会在主线程上被访问，外部调用者也可以从 API 里获得这些信息，这样可以保证可以在主线程执行。

MainActor 在程序中是一个全局的单例，他能保证所有的代码都能在主线程中更新，与普通 Actor 最大的不同就在于他始终运行在主线程。

```swift
@globalActor public actor MainActor {
public static let shared: MainActor
}
```

使用的方式也很简单，我们只需要将需要在主线程上执行的方法前面加上 `@MainActor`，就可以保证方法始终在主线程中执行。

```swift
@MainActor func runInmain(){
print("run in main")
}
```

MainActor 可以标记 class，让 class 中的属性，方法都在主线程上运行，这样避免了很容易被遗漏的到切换到主线程的操作。当然，单个方法也可以在其他线程上执行，用 `nonisolated` 去标注那些方法。

```swift
@MainActor class Count{
...
nonisolated func calculate() async { ... }
}
```

### 总结

这里你看到了如何用 Actor 优雅的解决线程数据竞争问题，并且在线程中用 Sendable 数据在多线程中安全的传递数据，避免了使用锁等其他工具的繁琐，最后 MainActor 保证了我们的代码是中运行在主线程。Actor 是 Swift 并发模型的核心部分，它们与 async/await 并发一起工作，让构建正确高效的并发程序变得更容易，期待用 Actor 去构建安全的高效的 Swift 代码。
