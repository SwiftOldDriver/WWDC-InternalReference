## 前言

Apple 致力于 Accessibility（无障碍辅助功能）并不是一天两天的事情，几乎每年都有超过 5 个 Session 在谈论 Accessibility 亦或是包容性（inclusive）；今年 WWDC 首次推出了四个 Digital Lounges，其中之一的主题就是 Accessibility。如果你持续关注 Accessibility 的进展，很容易会发现许多改变都发生于细微之处，我认为这恰恰正是 Apple 令人尊敬的地方，关注细节，不放弃任何用户，日复一日把事情做得更好。

今年，Apple 带来了全新的音频图表（[Audio Graphs](https://developer.apple.com/documentation/accessibility/audio_graphs)）来帮助我们提升图表的无障碍体验。不妨就以此为契机，一起来深入探索图表无障碍化的基础知识、现状和新特性。

## 图表无障碍支持的现状

图表非常有用，它被广泛运用在各个领域，基于其由大量视觉化符号和极少数文字构成的特质，我们往往可以很容易地从中快速获取大量信息，并轻松掌握数据之间的关系和变化趋势，说一图胜万言也不为过。然而，正如 Session 所说：“看不到图表，数据就不存在”，我们必须从头考虑如何让图表在用户缺乏视觉感知时也发挥威力。

如果你曾经涉足于图表的无障碍支持，或许会感到困惑：VoiceOver 不是已经可以帮助我们读出图表中的数据了吗？为什么还要在 WWDC 中旧事重提？倘若这是你第一次接触这个概念，我推荐在 Apple 的股市 App 中体验一下 VoiceOver 的杰出表现。然后你会发现，我们可以与图表进行交互，手指所到之处 VoiceOver 都会为我们读出来。简单来说，图表被划分为了多个区间，每个区间都代表一个可以被无障碍访问的数据点。

当前的图表无障碍支持程度是令人欣喜的，但并不完美，不妨来思考一下这两个问题：

- 如何快速了解一支股票的涨跌趋势？
- 如何快速找到该股票的历史最低点或历史最高点？

## 为图表赋予可操作性

尽管不完美，但聊胜于无，让我们暂时忘记上面的两个问题，先来了解一下现阶段该如何为图表赋予可操作性以便用户可以与之进行交互。接下来我们将围绕一张表示咖啡量（杯）与代码量（行）关系的柱形图展开讨论，不得不说，我十分喜欢 Accessibility 团队的例子，数据纯属虚构，但冷幽默让人无法拒绝。

![02-coffee-loc](https://cdn.nlark.com/yuque/0/2021/png/21889291/1624876009797-8153b5d2-ce62-479a-a021-a543f6ff71d8.png)

柱状图中的每一条立柱表示一个数据点，X 轴和 Y 轴的数据分别代表咖啡量和代码量。正如上一个部分提到的，图表由众多数据点构成，所以首先需要在 `ChartModel` 中定义数据点的集合：

```swift
class ChartView: UIView {
    let model: ChartModel
}

struct ChartModel {
    let title: String
    let dataPoints: [DataPoint]
        
    struct DataPoint {
        let name: String
        let x: Double
        let y: Double
    }
}
```

之后，便可以开始着手为 `ChatView` 实现无障碍支持。为了简化场景，假设上述的柱状图是基于 Core Graphics 绘制而成，也就是说，柱状图内部的所有元素都不是 `UIView` 的子类，只是一些线条和图形。显然，它们不会自动获得无障碍属性。

因此，在正式开始实现无障碍支持之前，还需要了解一个名为 [`UIAccessibilityContainer`](https://developer.apple.com/documentation/objectivec/nsobject/uiaccessibilitycontainer) 的非正式协议，它与 `UIAccessibility` 一样，都是 UIKit Accessibility 最重要的组成部分。这个协议使得 `UIView`（或它的子类）所包含的任意对象可以被作为独立个体单独访问，即便那些对象与 `UIKit` 毫无关系，这样一来，我们便可以将柱状图中的每个柱体当作抽象的视图并为之赋予无障碍属性。

让我们开始吧！首先，要明确 `ChatView` 的容器类型。这是一个 `UIAccessibilityContainerType` 类型的枚举值，其中一些成员只能用于特定容器（该枚举从 iOS 11 开始被导入，你可以在 [WWDC2017 Session 215](https://developer.apple.com/videos/play/wwdc2017/215) 中找到当时的介绍）。根据定义，为了让 VoiceOver 帮助我们读出包括 `accessibilityLabel` 在内的设置在容器上的无障碍属性 ，需要将容器类型设为 `semanticGroup`，如下所示：

```swift
@available(iOS 11.0, *)
public enum UIAccessibilityContainerType : Int {
    case none = 0
    case dataTable = 1 // 使用该类型的同时务必实现 UIAccessibilityContainerDataTable 协议
    case list = 2
    case landmark = 3
    
    @available(iOS 13.0, *)
    case semanticGroup = 4 // 辅助功能会查询设置在容器上的无障碍属性，例如accessibilityLabel，以便向用户输出相关信息
}

extension ChartView {
    public override var accessibilityContainerType: UIAccessibilityContainerType { 
        get {
            return .semanticGroup
        }
        set { }
    }
}
```

然后，我们需要为图表设置一个无障碍标签，以告诉 VoiceOver 当聚焦到图表时该做出什么样的描述。通常返回标题即可，当然也可以是其他简短的文字。但要注意，它需要有独特性，让用户能够第一时间知道自己正在阅读什么：

```swift
extension ChartView {
    public override var accessibilityLabel: String? { 
        get {
            return self.model.title
        }
        set { }
    }
}
```

最后也是最重要的一步，让我们来把数据点一一转换为支持无障碍访问的元素。这并不复杂，一方面我们需要为无障碍数据点设置一个经过本地化的字符串，这样一来，当聚焦到该数据点时 VoiceOver 便会为我们读出当前值，例如：2 杯咖啡 20 行代码；另一方面，因为柱体只是虚拟的视图，所以还需要为它设置一个相对的容器的 `frame`，这样 VoiceOver 才会知道这些柱体的位置和尺寸：

```swift
extension ChartView {
    public override var accessibilityElements: [Any]? {
        get {
            return model.dataPoints.map { point in
                let axElement = UIAccessibilityElement(accessibilityContainer: self)
                axElement.accessibilityValue = "\(point.x) cups, \(point.y) lines of code"
                axElement.accessibilityFrameInContainerSpace = CGRect(...)
                return axElement
            }
        }
        set {}
    }
}
```

是不是很简单？只需设置屈指可数的几个无障碍属性就能够获得这么多的好处。这里还有一个小建议：以股市 App 为例，以日为单位的话，一支股票上市超过一年会有超过 200 个数据点，将每个数据点都转化为无障碍元素是不切实际的。合理地将数据点分组，以组的名义来创建无障碍元素是非常机智且有必要的。

## 音频图表

### 数据可听化（Data Sonification）

还记得前面提过的两个问题吗，你是否有了什么头绪？乍一看，除非添加额外的功能来帮助获取趋势和极值点，否则耐心听完所有可用数据应该是唯一的方法。而音频图表的诞生，能够让我们在解决这些问题的路上迈出跨越性的一步。

让我们先从一个简单的例子开始吧。Accessiblity 团队提供了一个表示年度出生率的折线图案例，通过[转子]((https://support.apple.com/zh-cn/HT204783))选中音频图表并切换到图表详细之后，可以播放出生率的变化趋势，你没有看错，趋势和声音这两个原本毫不相关的概念竟然被结合到了一起。原理在于，VoiceOver 会将 Y 轴的数值大小转换为音调的高低，这样一来，当听完图表的音调变化之后，用户就可以很清晰地知道出生率的变化趋势了。

![03-birth-rates](https://cdn.nlark.com/yuque/0/2021/png/21889291/1624876180960-9c4d8437-44da-4d52-bf10-9410a69b6a40.png)

关于第二个问题，如何获取最大值呢？事实上，音频图表还提供了交互模式，双击图表，长按并沿 X 轴移动手指，在听到最高音调时停下来，稍微等待，VoiceOver 就会为我们读出当前坐标所对应的年份和出生率值。

音频图表非常有用，不过这并不是一个旷古绝今的创造，它是数据可听化的优秀实践。数据可听化这个概念贯穿我们生活的点点滴滴，除了音调之外，常常还会用到音量，音色，音长等特征。

### 图表的概要与特征

你或许已经注意到，在音频图表的下方出现了概要（Summary）和特征（Feature）。这些文字描述是 Apple 基于图表的数据和特征自动生成的，其中了包含图表的形状、趋势和异常值等。尽管这个功能目前只被轻描淡写地提起，我相信随时光更迭，它必将越来越有用。

## AXChart Protocol

在了解音频图表之后，改进上述咖啡 vs 代码柱状图的机会终于来了！简直令人迫不及待。

首先，我们需要为 `ChartModel` 添加几个新属性，从上一小节的叙述中可以推测出来，音频图表在数据点之外，更多地关注了图表整体。因此，需要添加几个有关坐标系和概要信息的属性：

```swift
struct ChartModel {
    let title: String
    let summary: String
    let xAxis: Axis
    let yAxis: Axis
    let data: [DataPoint]

    struct Axis {
        let title: String
        let range: ClosedRange<Double>
    }
    
    struct DataPoint {
        let name: String
        let x: Double
        let y: Double
    }
}
```

然后，让我们来着手为 `ChatView` 提供音频支持。第一步，让 `ChatView` 遵循 `AXChart` 协议，这并不困难，需要实现的属性仅有 `accessibilityChartDescriptor`：

```swift
import Accessibility

extension ChartView: AXChart {
    public var accessibilityChartDescriptor: AXChartDescriptor? {
        get { }
        set { }
    }
}
```

`accessibilityChartDescriptor` 的返回值类型是 `AXChartDescriptor`，它有两个初始化方法，我们需要使用的是：`init(title:summary:xAxis:yAxis:additionalAxes:series:)`。接下来，让我们一起来看看每个初始化参数究竟意味着什么。

`title` 和 `summary` 很显然是自定义标题和概览，它们都将会出现在音频图表的视图中，此处就不再赘述。

`xAxis`和 `yAxis` 分别是两个坐标轴，它们共同定义了坐标系；每个坐标轴都是一个遵循 `AXDataAxisDescriptor` 协议的对象，只不过，X 轴既可以是 [`AXCategoricalDataAxisDescriptor`](https://developer.apple.com/documentation/accessibility/axcategoricaldataaxisdescriptor) 也可以是 [`AXNumericDataAxisDescriptor`](https://developer.apple.com/documentation/accessibility/axnumericdataaxisdescriptor)，而 Y 轴代表着音调的高低，因此只能是用于表示数字的 `AXNumericDataAxisDescriptor`。回到我们的例子中，由于咖啡量和代码量都是具体数字，所以两条坐标轴都应当是 `AXNumericDataAxisDescriptor` 对象。另外，如果图表有网格线（Grid Line），则需要向 `gridlinePositions` 传递一个 `Double` 类型的数组。最后，为了让 VoiceOver 读出人类可以理解的内容，还需要为数据点上的数值加上单位和合适的描述。例如，X 轴上的数据最好是“五杯咖啡”而非“五”：

```swift
public var accessibilityChartDescriptor: AXChartDescriptor? {
    get {
        let xAxis = AXNumericDataAxisDescriptor(
            title: model.yAxis.title,
            range: model.yAxis.range,
            gridlinePositions:[],
            valueDescriptionProvider: { value in
                "\(value) cups"
            }
        )
        let yAxis = AXNumericDataAxisDescriptor(
            title: model.yAxis.title,
            range: model.yAxis.range,
            gridlinePositions:[],
            valueDescriptionProvider: { value in
                "\(value) lines of code"
            }
        )
    }
    set {}
}
```

坐标系的构建完成之后，我们将引入数据系列（Data Series）的概念来描述数据点，与之对应，这一步需要完成对 `AXDataSeriesDescriptor` 对象的初始化。通常，一组相关数据点构成的集合可以被称为数据系列；反过来说，一个数据系列主要包含数据点和该数据集的连续性。这里的连续性，对于柱状图和点状图而言显而易见是非连续的，而折线图则是连续的，`isContinuous` 代表了该属性。最后，再将数据点数组也作为参数传入，我们就得到了一个数据系列。注意，一个图表中可以包含不止一个数据系列，不过在我们的例子中，仅有一组数据：

```swift
public var accessibilityChartDescriptor: AXChartDescriptor? {
    get {
        let xAxis = AXNumericDataAxisDescriptor( … )
        let yAxis = AXNumericDataAxisDescriptor( … )
        let series = AXDataSeriesDescriptor(
            name: model.title,
            isContinuous: false,
            dataPoints: model.data.map { point in
                AXDataPoint(
                    x: point.x,
                    y: point.y,
                    additionalValues: [],
                    label: point.name
                )
            }
        )
    }
    set {}
}
```

一切准备就绪，完成 `AXChartDescriptor` 的初始化，一张支持音频的图表就大功告成了：

```swift
public var accessibilityChartDescriptor: AXChartDescriptor? {
    get {
        let xAxis = AXNumericDataAxisDescriptor( … )
        let yAxis = AXNumericDataAxisDescriptor( … )
        let series = AXDataSeriesDescriptor( … )
        return AXChartDescriptor(
            title: model.title,
            summary: model.summary,
            xAxis: xAxis,
            yAxis: yAxis,
            additionalAxes: [],
            series: [series]
        )
    }
    set {}
}
```

到这里，有关音频图表的讲解就告一段落了，我们不妨来回顾一下这跨越性的一步是如何实现的。我认为最重要的有两点，一是借助音调来呈现数值，二是把关注点从各个独立数据点扩大到坐标系整体。

## 谈一谈视觉上的无障碍图表设计

在讨论了很多高阶的内容之后，让我们回过头来重新谈一谈无障碍图表设计的基础，千里之行，始于足下。在视觉障碍的世界中，除了视力缺失，还存在各种程度的弱视和各种类型的色盲，也就是说，在支持无障碍的路上还充满着许多未知。

事不宜迟，首先让我们来看看下面这张热带和沙漠的月均降水量折线图，它包含多个致命的设计问题，你能发现几个呢？

### 使用高对比度的颜色

对于弱视群体而言，上图存在着显著的对比度过低的问题，我们可以通过逐渐调暗屏幕亮度，来感受两条折线的颜色与背景越来越难以区分的过程。

或许我们很难留意到这些问题，所幸，在 Accessibility Inspector 中集成了一个可以帮助我们判断两种颜色的对比度是否合理的工具，你可以通过 Accessibility Inspector -> Window -> Show Color Contrast Calculator 找到它。通常，对于较小的字体而言，`4.5 : 1` 是对比度的最低限。关于颜色对比度的问题，HIG 也为我们提供了指南：[Color and Contrast](https://developer.apple.com/design/human-interface-guidelines/accessibility/overview/color-and-contrast/)。

### 常见的问题颜色搭配

红绿色盲者眼中的降水量折线图实际是下面这样的，过于近似的颜色导致图表完全失去了本该有的含义：

一个简单的解决方式是将红色改为蓝色：

红绿，蓝黄是最普遍的两种色盲类型，因此我们应该尽可能避免使用两种颜色搭配。

### 为图表加上颜色之外的特征

除了针对颜色的考量，我们可以换个思路，为图表加上符号，这样一来，用户可以通过识别符号而非单纯依赖颜色来获取图表信息。

![09-symbol](https://cdn.nlark.com/yuque/0/2021/png/21889291/1624877340016-b2a36532-4214-40bc-8294-28e21dcdaee1.png)

这里提供一个小 Tip，除了符号之外，在折线图中还可以使用实线和虚线来区分不同的数据系列。而对于其他类型的图表，我们也一样可以为它们加上除了颜色以外的形状特征来帮助理解。

### Accessibility 设定

有时候，产品的设计会限制我们对颜色的选择，或者不能轻易为图表加上符号或形态特征。但是没关系，我们依然可以通过支持 Accessibility 的设定来为图表提供无障碍的体验。

![10-settings](https://cdn.nlark.com/yuque/0/2021/png/21889291/1624877358292-77d650fa-d52a-43f5-a41e-b6fec4cc91e8.png)

以上图中的“（Differentiate Without Color）不以颜色区分”为例，它对应 [`shouldDifferentiateWithoutColor`](https://developer.apple.com/documentation/uikit/uiaccessibility/3043553-shoulddifferentiatewithoutcolor) 属性，仅当该属性为 `True` 时，为图表加上颜色之外的特征来帮助用户理解图表也不失为一个好办法。我们可以在 [UIAccessibility#Capabilities](https://developer.apple.com/documentation/uikit/uiaccessibility) 中找到更多这样的设定。

## 尾声

通过这篇文章，我们从数据可视化迈入了数据可听化的世界。来重新梳理一下本文吧，简而言之，如果要打造一张接近完美的无障碍图表，要做的事情其实只有以下三件：

1. 关注颜色的搭配和对比度，在可能的情况下为图表加上符号和形态特征。
2. 为图表赋予可操作性，谨记“看不到图表，数据就不存在”。
3. 支持音频图表，帮助用户更迅速地理解图表的变化趋势和极值。

举手之劳，也许就会成为一束光，照亮某个人的世界。期待更多开发者能够多投入一些精力，去思考如何提供更友好且高效的无障碍支持。
