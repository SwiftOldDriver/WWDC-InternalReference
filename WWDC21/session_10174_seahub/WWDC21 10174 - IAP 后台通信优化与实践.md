### JWS 票据校验 —— 使用依赖证书链的自校验方式进行票据校验

针对支付领域，我们一般将支付票据定义为：可以用于校验某笔交易是否合法的数据。针对上定义，再结合 IAP 场景，目前可以被认为是票据的字段有属于有以下三类：

* [transactionReceipt](https://developer.apple.com/documentation/storekit/skpaymenttransaction/1617722-transactionreceipt)：始于 iOS 3，但在 iOS 7 被标记为废弃的老票据
* [App Receipt](https://developer.apple.com/documentation/foundation/bundle/1407276-appstorereceipturl)：始于 iOS 7，并一直沿用至今的票据
* StoreKit 2 Transaction.jws（下简称 JWS Receipt）：WWDC21 发布，iOS 15、StoreKit 2（Swift Only）可用

三种票据简单对比分析可得出如下表格

|                                           |                  transactionReceipt                  |                         App Receipt                          |              JWS Receipt               |
| :---------------------------------------: | :--------------------------------------------------: | :----------------------------------------------------------: | :------------------------------------: |
|               **获取时机**                |                   从支付回调中获取                   | 可以任意时候获取，但有可能获取到缓存票据或获取失败；若想获取最新有效票据，需要用户密码授权 |           大部分情况下可获取           |
|               **获取速度**                |                         较快                         |                             较慢                             |                  较快                  |
|               **票据大小**                |                    较小（约 4KB）                    |                       较大（约 123KB）                       |             较小（约 2KB）             |
|               **交易信息**                |                      仅单笔交易                      |  包含最近 100 笔自动续期订阅型、非消耗型、非续期订阅型交易   |               仅单笔交易               |
|               **信息内容**                |                      仅交易信息                      |   除交易信息外，还有额外的信息，如预订信息、应用基本信息等   | 包含交易信息、签名及签名用的算法证书链 |
| **是否需要带密钥请求  VerifyReceipt API** | 仅自动续期订阅型**需要**带密钥请求 VerifyReceipt API |       仅非消耗型**不需要**带密钥请求 VerifyReceipt API       | **无需请求 VerifyReceipt API，自校验** |

注意上述表格的 JWS Receipt 的两点特性：

* 自校验：目前无论是 transactionReceipt 还是 App Receipt，都是需要使用 VerifyReceipt API 进行票据校验。而对于 JWS Receipt，我们可以无需再依赖不稳定且请求时耗长的苹果 VerifyReceipt API 进行票据有效性校验，这个特性对我们而言，是极为重要的
* 票据大小：由于票据内容重新回归至单笔交易，所以票据大大减少，我们再也不用担心票据出现传输超时的情况

那么问题来了，我们怎么样校验 JWS Receipt 呢？解答之前，我们先来了解一下 JWS Receipt 格式。其以逗号分隔为三部分：由校验算法及用于校验签名的证书链组成头部；由交易信息组成内容部分；由签名作为其最后一部分，可参考如下案例：

```
// Base64(header) + "." + Base64(payload) + "." + sign(Base64(header) + "." + Base64(payload))

// header
{
  "alg": "ES256", // 校验算法
  "x5c": [...]    // 用于校验签名的证书链
}

// Payload
{
  "transactionId": "1098836",
  "originalTransactionId": "1082212",
  "webOrderLineItemId": "65738567439",
  "bundleId": "co.ocenjournal",
  "productId": "co.ocenjournal.monthlySubscription",
  "subscriptionGroupIdentifier": "12345",
  "purchaseDate": 1623081600000,
  "originalPurchaseDate": 1623081600000,
  "expiresDate": 1625673600000,
  "quantity": 1,
  "type": "Auto-Renewable Subscription",
  "appAccountToken": "37e0a95b-4455-42e6-bac2-e59259c8aac2",
  "revocationDate": 162515520000,
  "revocationReason": 0,
  "offerType": 2,
  "offerIdentifier": "co.ocenjournal.offer1"
}
```

结合上述结构，我们可通过以下步骤进行自校验：

1、从 Header alg 部分获取校验算法

2、从 Header x5c 部分获取用于校验的证书链

3、结合获取的校验算法、证书链，使用合适的第三方库对 Sign 部分进行校验

4、校验成功后，读取 Payload 部分

需要注意，相比以往使用 App Receipt 调用 VerifyReceipt API 校验获取的 Payload，JWS Receipt Payload 有以下的改变：

- 命名规范修改
	- 使用小驼峰风格替代下划线风格
	- 时间相关的字段均采用 UNIX 时间戳格式（以往直接返回 GMT 字符串格式）
	- 部分字段名称修改（cancellation_date => revocationDate、cancellation_reason => revocationReason）
- 新增类型字段 type 表示苹果物品类型，不过比较蛋疼的返回的是一个字符串，而不是类似 OfferType 的枚举值

- 新增透传字段 appAccountToken 表示用户账号唯一映射的 UUID（App Receipt 每个 Transaction 中同样会增加相应字段）（[通过 StoreKit 2 可传入该字段](https://developer.apple.com/documentation/storekit/product/purchaseoption/3749440-appaccounttoken)）

- 新增字段 offerType，表示自动续期订阅促销类型（以前有 is_trial_period、is_in_intro_offer_period 等类似作用的字段，现在相当于将自动续期订阅促销相关的字段聚合起来，更加合理），取值范围为 [1, 3]
	- 1：intro offer，该笔购买使用了[推介促销](https://developer.apple.com/documentation/storekit/original_api_for_in-app_purchase/subscriptions_and_offers/implementing_introductory_offers_in_your_app)，即 AppleID 首开某个订阅组内的新自动续期订阅可享受的折扣活动
	- 2：promotional offer，该笔购买使用了[促销优惠](https://developer.apple.com/documentation/storekit/original_api_for_in-app_purchase/subscriptions_and_offers/implementing_promotional_offers_in_your_app)，即 AppleID 第 N + 1 次开通某个订阅组内的自动续期订阅可享受的折扣活动
	- 3：offer code，该笔购买使用了[优惠代码](https://developer.apple.com/documentation/storekit/original_api_for_in-app_purchase/subscriptions_and_offers/implementing_offer_codes_in_your_app)，即使用优惠代码对自动续期订阅进行了折扣购买

- 新增字段 offerIdentifier，表示促销优惠 / Offer Code 的优惠标示（以前有 promotiaonal_offer_identifier、offer_code_ref_name，现在该字段同样是将以前的多个字段聚合起来）



### Subscription Status API —— 获取自动续期订阅最新状态

目前针对自动续期订阅，我们可以通过后台通知获取最新状态。但对于没有接入后台通知的业务，我们目前只能定期进行扫表处理（即通过扫描业务订单库，在自动续期订阅将要到期时的前后 3 天 ，调用 VerifyReceipt API 获取最新状态）。从业务需求上来看，目前的技术上是已经满足的。但从技术实现上，业务订单往往数量众多，而 VerifyReceipt API 请求时耗又高，这就导致扫表过程缓慢，需要定期分批次进行处理

另外，由于新的 JWS Receipt 仅包含当前交易的状态，苹果需要通过新的接口来告知我们一些订阅相关的信息，同时也能降低我们对 VerifyReceipt API 的依赖度。因此，苹果在 WWDC21 为我们提供了 Subscription Status API。通过该特性，我们可以改用该接口，对自动续期订阅的最新状态进行获取

- Subscription Status API 需要使用 JWT 校验方式（可参考下文内容）并提供订单唯一标示 {original_transaction_id}
	- API：[/inApps/v1/subscription/{originalTransactionId}](https://developer.apple.com/documentation/appstoreserverapi/get_all_subscription_statuses?changes=la)

### In-App Purchase History API —— 获取用户历史购买记录

目前如果我们想要获取用户购买的历史，只能通过使用 App Receipt 请求 VerifyReceipt API 获取 in_app / 自动续期订阅特有的 latest_receipt_info 数组，然后对其进行扫描。类似的，这里会存在对 VerifyReceipt API 的依赖，导致时耗增加。除此之外，这里还存在一个问题，根据苹果文档描述，in_app / latest_receipt_info 数组包含了最近 100 笔非消耗型、非续期订阅型、自动续期订阅型的用户购买历史记录。100 笔对于我们来说，太多也太少。另外不能获取消耗型购买记录，对于我们也是一个很大的痛点

* 太多：一个 App Receipt 内假设包含 100 笔请求，那么其大小往往也很大，大概率为 100KB 以上。且对于大票据，苹果校验票据的时耗也会增长，往往为 5s 以上
* 太少：我们希望的是获得**所有**的历史购买记录，若用户进行了 100 + N 笔的购买，那么我们就会丢失 N 笔的历史记录

另外，由于新的 JWS Receipt 仅包含当前交易的状态，苹果需要让使用 JWS Receipt 的业务也能够获取到历史记录。因此苹果新增了 In-App Purchase History API，用于供我们获取用户的历史购买记录。该接口除了能够有效支持 JWS Receipt 外，还能够帮我们解决以上场景的痛点

* In-App Purchase History API 需要使用 JWT 校验方式（可参考下文内容）并提供订单唯一标示 {original_transaction_id}

- API：[/inApps/v1/history/{originalTransactionId}](https://developer.apple.com/documentation/appstoreserverapi/get_transaction_history?changes=la)
- 调用结果如下图示：

![3.png](https://cdn.nlark.com/yuque/0/2021/png/496659/1624816482796-13662cac-cfbe-4023-aded-6752c3674cae.png?x-oss-process=image%2Fresize%2Cw_1503)

注意：该接口支持分页功能，signedTransactions 包含 20 笔最新购买的 JWS，若还有更多的内容时，此时字段 hasMore == true。我们若要获取更多的历史记录，需结合 revision 字段发起请求，即以 /inApps/v1/history/{originalTransactionId}&revision={revision} 格式发起请求

1、从 App Store Connect 配置 **In-App Purchase Keys**（参考下图）（没错，又是一个新的 Key，**截止目前，已有共享密钥、App 专用共享密钥、促销优惠密钥、In-App Purchase Keys 等四类 IAP 相关的密钥**）

![4.png](https://cdn.nlark.com/yuque/0/2021/png/496659/1624816841263-4dacae4f-823d-4a70-9e2f-864528fb949e.png?x-oss-process=image%2Fresize%2Cw_1503)

2、从 App Store Connect 中获取 KeyID，用于放置 JWT header.kid 字段

3、从 App Store Connect 中获取 IssuerID，用于放置 JWT payload.iss 字段

4、使用 ES256 算法及 In-App Purchase Keys 私钥对请求提进行签名

5、将头部、内容部分及签名部分通过逗号拼接，获得 JWT，用于请求相关 API

![5.png](https://cdn.nlark.com/yuque/0/2021/png/496659/1624816841937-909b7037-379b-40a5-89cb-4af6ddd3bbff.png?x-oss-process=image%2Fresize%2Cw_1503)

注意：签发时间 iat 与过期时间 exp 相隔必须在一小时内



### 后台通知升级（App Store Server Notification V2）

众所周知，苹果 IAP 后台通知设计一直以来都饱受诟病，用户主要吐槽的点有两个：

* 用户进行一个事件可以导致多个通知发送。比如用户执行同一订阅组的升降级切换，苹果会发送 DID_CHANGE_RENEWAL_STATUS、INTERACTIVE_RENEWAL 等通知（注意，在 2020 年末前，订阅升级事件会发送 DID_CHANGE_RENEWAL_STATUS、INTERACTIVE_RENEWAL。但在 2020 年末，苹果针对升级事件取消了 CANCEL 通知的发送）
* 部分场景仍有覆盖不到的场景，如 Grace Period 是否过期（根据苹果文档，用户处于 Grace Period 时，我们应该提供有限的服务，当用户完全过期后再进行服务停止，所以我们迫切的希望知道 Grace Period 是否过期）。目前我们只能通过 DID_FAIL_TO_RENEW 获取 grace_period_expires_date 配以 VerifyReceipt API 进行间接处理；再如以往没有办法感知 Offer Code 使用等事件

通过上述表格，我们可以看到当前通知种类繁多，且一个用户场景可能会触发多种通知，导致我们的处理逻辑需要编写的极为复杂。苹果似乎也感受到了我们的痛点，终于在今年发布了 App Store Server Notification V2。其遵循"对于每个事件，通知仅发一个"的原则进行优化

#### App Store Server Notifications V2（WWDC21，年底上线）

通知类型发生了以下变化

- 废弃 4 类：INITAL_BUY、INTERACTIVE_RENEWAL、CANCEL、PRICE_INCREASE_CONSENT
- 新增 4 类：SUBSCRIBED、OFFER_REDEEMED、EXPIRED、GRACE_PERIOD_EXPIRED、PRICE_INCREASE

- 同时，在 [10175 - Support customers and handle refunds](https://developer.apple.com/videos/play/wwdc2021/10175) 中，有额外新增两类通知类型：CONSUMPTION_REQUEST、REFUND_DECLIEND

### 家庭共享通知优化

苹果今年也对家庭共享相关的通知进行了优化，当前家庭共享支持与家庭分享自动续期订阅类型及非消耗型类型。目前家庭成员做相关的订阅处理时，我们也会收到相应的[通知](https://developer.apple.com/documentation/appstoreservernotifications/notification_type)（REVOKE、DID_RECOVER、DID_FAIL_TO_RENEW）。显然，虽然能够收到通知，但是少了很多类型的通知，导致我们没有办法去完全追踪家庭成员订阅状态，也可能存在灰产分享的漏洞



于是，今年苹果也对家庭共享场景通知进行了调整，以试图覆盖更多的场景

针对 V1 家庭共享场景下的订阅通知，新增以下类型（购买者及家庭成员执行相应事件时，均会发出通知），完整类型可参考上文的表格：

- DID_CHANGE_RENEWAL_STATUS
- DID_CHANGE_RENEWAL_PREF

- DID_RENEW
- INTERACTIVE_RENEWAL

针对 V2 家庭共享场景下的订阅通知，新增以下类型（购买者及家庭成员执行相应事件时，均会发出通知），完整类型可参考上文的表格：

- DID_CHANGE_RENEWAL_STATUS
- DID_CHANGE_RENEWAL_PREF

- DID_RENEW
- SUBSCRIBED

- EXPIRED
- GRACE_PERIOD_EXPIRED

- OFFER_REDEEMED



### 沙箱测试优化

长期以来，我们在开发阶段，想要系统测试自动续期订阅的场景，往往需要配置若干个苹果沙箱测试账号，其根本原因在于我们没办法清空苹果沙箱测试账号的购买记录，且一个账号仅能在创建时选择地区以测试外币

终于，苹果今年回应了我们的诉求。我们可以在 App Store Connect 中对沙盒测试账号历史购买内容进行清理，同时也可以动态改变现有沙盒测试账号的区域，甚至还可以调整在沙盒中的自动续期频率

## 结语

总的来说，苹果这次针对开发者反馈的两大块问题做了重点优化

* 票据校验时耗及过于依赖 VerifyReceipt API 的问题：新增 Subscription Status API & In-App Purchase History API 用以对部分场景进行分流处理。同时对于 JWS Receipt，我们可以不再依赖苹果，直接进行自校验。我们可以基于新增的接口对部分场景进行优化
* 后台通知难用的问题：升级通知，新增 App Store Server Notification V2。我们可以基于新的通知重新设计用户追踪路径
