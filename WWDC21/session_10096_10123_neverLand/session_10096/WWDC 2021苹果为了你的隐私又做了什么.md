# WWDC 2021 苹果为了你的隐私又做了什么

> 作者：刘杰，目前就职于人民日报媒体技术股份有限公司，负责融媒云项目开发。
>
> 审核：郭鹏，老司机技术周报编辑，就职于丁香园，丁香妈妈 App 开发。

基于 [Session 10096](https://developer.apple.com/videos/play/wwdc2021/10096?time=29) 梳理。

> 众所周知，苹果一直着力重视用户隐私，近几年在 WWDC 上不断退出新的隐私特性。在今年 WWDC 2021，苹果围绕用户隐私推出了一系列创新功能：隐藏邮件地址、家庭安全监控和 iCloud Private Relay。这篇文章主要介绍 iCloud Private Relay。

![](https://images.xiaozhuanlan.com/photo/2021/387a4cdc22e79944460b6857f5e8ec1a.png)

**iCloud Private Relay（ iCloud 私人中继）**是苹果新推出的网络隐私功能，它可以阻止网络或者服务器监控用户行为、对网络传输进行加密。作为 iCloud+ 订阅的一部分，iCloud Private Relay 可以在用户浏览网页和使用 APP 时，防止不经意间泄露个人信息或置身于安全攻击中。

![](https://images.xiaozhuanlan.com/photo/2021/4aaa39f26dd00c5d5b25e8eb8985d59d.png)

- 什么是 iCloud Private Relay？
- 它对 APP 会有什么影响?
- 网站和服务器需要做哪些适配工作？
- 在 Private Relay 生效时如何管理网络并监控传输?

带着这些疑问，接下来就来了解一下苹果在本届开发者大会新推出的网络隐私特性 —— iCloud Private Relay。

## 什么是 iCloud Private Relay

### Private Relay 介绍

iCloud Private Relay 内置在 iOS 和 macOS 系统中，这样你不需要为你的 APP 做任何事情就可以使用它。但是，需要注意的是它并不是持续影响你的应用，只有在用户订阅了 iCloud+ 并且开启 Private Relay 权限时才能生效。

你可能好奇：Private Relay 做了什么？我们先了解一下现在网络传输过程存在的问题。

![Without iCloud Private Relay](https://i.loli.net/2021/06/16/EhnWvPXsmziqDxQ.png)

 1. 当我们访问网络，在本地网络的任何人都可以通过检查 DNS 查询记录看到我们访问的所有网站的名称。这个信息可以用来获取用户的浏览器“指纹”并建立访问历史记录。无论是公共 Wi-Fi 的运营商、网络中的其他用户，亦或者网络服务器供应商都不应该有悄无声息地收集用户访问的网站信息能力。
2. 当网络访问连接到运行网站的服务器，服务器就知道了访问用户的 IP 地址，这将使服务器有能力在没有明确用户同意的情况下判断用户位置。更糟糕的是：即使使用像 Sarari 中的 Intelligent Tracking Prevention (智能反追踪)工具阻断 Cookie 关联性，服务器还是可以识别用户身份、识别不同网站的用户。

这些都是很严重的用户隐私问题，为了摆脱这些问题，我们需要一种从设计之初就内置隐私处理的新方法 —— iCloud Private Relay。iCloud Private Relay 增加了多重安全代理帮助用户控制传输路由且保证其私密性。这些代理由独立的实体运行，其中 Ingress Proxy 是由苹果提供，Egress Proxy 是由内容提供商提供。

![With iCloud Private Relay step 1](https://i.loli.net/2021/06/16/kqHPCd3pJTy2W5f.png)
![With iCloud Private Relay step 2](https://i.loli.net/2021/06/16/p3jlJuiFRaEAU7m.png)


当 iCloud Private Relay 生效时，客户端 IP 地址只对网络服务商和 Inpress Proxy 是可见的。Egress Proxy 只能获取到用户当前请求的内容，然后使用它去建立和服务器的连接。需要注意的是，在这个过程中，Ingress Proxy 与 Egress Proxy 相互独立，没有任何节点（包括苹果公司）可以同时获取客户端 IP 地址和用户正在访问的内容。这样，用户完整信息构建的隐患就被消除了。

![](https://images.xiaozhuanlan.com/photo/2021/6ec9fa1337a120fc354dd45b4a0a7df3.png)

有一些点需要注意

1. 在 iOS 15 和 macOS Monterey 中, Private Relay 会适用于所有的 Safari 网页浏览、所有的 DNS 名称解析查询以及一小部分来自于 APP 的网络传输。特别需要注意的是，这将包含所有不安全的 HTTP 传输，例如通过 TCP 80 端口的网络传输。
2. 如果你的 APP 提供了内容过滤或者家长控制，网络传输在通过 Private Relay 之前仍可被查看，所有你提供的过滤不会受到任何影响。你可以在 [ "Meet the Screen Time API" session](https://developer.apple.com/videos/play/wwdc2021/10123/)([《【WWDC21 10123】初见 Screen Time API》](https://xiaozhuanlan.com/topic/6874305291)) 了解更多。
3. 应用中的网络连接不是都在公共网络发生，所有有几种类型的网络传输是不受 Private Relay 影响的：
 - 应用中任何通过本地网络或者私有域名的连接都不会受影响。
 - 应用提供了网络拓展来添加VPN或应用代理能力，使用拓展和代理的传输不会受影响。
![](https://images.xiaozhuanlan.com/photo/2021/8d2677cc4a264da3b2045fe9a27ea07d.png)

 


### Private Relay 中的网络访问管理原理
为了确保传输的安全和高效，Private Relay 使用最新的传输代理和隐私保护身份验证。

![](https://images.xiaozhuanlan.com/photo/2021/6c1f8539820d323e46bd80e06905827e.png)

Private Relay 使用 `RSA Blinded Signatures` 管理网络访问权限，在建立连接之前，`Private Relay Access Token Server` 会给设备提供一组不同的 Token 用来支持选择不同代理供应商。接下来设备会运行一个盲加密操作将 Access Token 转化为 Unlinkable Token， Private Relay Proxy 使用 Token Server 提供的公钥来进行解密，快速的验证网络访问权限。

![](https://images.xiaozhuanlan.com/photo/2021/57c1a72238a9de718365583053f6a4af.png)

为了不同代理之间的信息分离，网络访问连接使用了多层加密，代理之间逐层解密，只有设备拥有解密所有层级的能力。Private Relay 的构建借鉴了 **Apple's privacy pillars** 的架构，更多内容详见 [ “Apple's privacy pillars in focus” session ](https://developer.apple.com/videos/play/wwdc2021/10085)（[《【WWDC21 10085】聚焦苹果隐私原则》](https://xiaozhuanlan.com/topic/6921584730)）。



## 如何使用 Private Relay
### 客户端准备工作
对于大部分应用，是不需要做任何事情的，Private Relay 会正常运行。不过，有许多很棒的实践还是需要了解一下。

#### 网络监控
无论你正在使用的是什么网络 API ，Private Relay 都会运行。几年来，苹果一直建议应用使用最新的 API ，例如 `URLSession` 和 `NWConnection` 。这些 API 是了解“ Private Relay 如何处理网络传输”最好的工具。对于 `URLSession` 可以使用 `Network Xcode Instrument` 来监测网络任务，即使是这些网络任务是通过 Private Relay 处理的也可以监测，详细的内容在 [ Analyze HTTP traffic in Instruments" session ](https://developer.apple.com/videos/play/wwdc2021/10212/) ([《【WWDC21 10121】为内容丰富的应用量身打造高效 VoiceOver 体验》](https://xiaozhuanlan.com/topic/7392456081))。

当 Private Relay 生效时，可以使用 `URLSession` 和 `Network.framework` 中相关的监测 API 来对其进行观察。 通过 `TaskTransactionMetrics` 可以检查任务是否使用了代理。通过 `NWConnection.EstablishmentReport` 可以检查 DNS 域名解析时间和代理连接的每个阶段。

#### 升级 HTTP 
相较于未使用加密策略而不安全的 HTTP 网络连接，Private Relay 迈出了很大一步。当 Private Relay 生效时，通过 TCP 80 端口访问的 HTTP 连接将被 Private Relay 代理，防止客户端与代理之间的本地网络受到攻击。虽然在 Private Relay 生效时能够保护网络安全，但这终究是治标不治本，追根溯源还是要将 HTTP 替换为 HTTPS来根本的解决问题。在项目中， `Info.plist` 中可能会设置一些支持 HTTP 连接的白名单，可以从这里开始入手进行 HTTP 的替换工作。
![](https://images.xiaozhuanlan.com/photo/2021/8b91f9fb95528548fbd7ba60ac38918a.png)

#### 更好的地理位置获取方式
如果应用提供了基于地理位置的功能，这也是个很好的机会来确认你正在使用 `Core Loacation` 的 API 而不是依赖于服务器从 IP 地址推断位置。IP 地址定位通常是不可靠的。`Core Location` 允许你指定你需要的定位精度并且它是基于明确的用户授权。


如果你确认应用已经准备好使用 Private Relay ，登录一个包含 iCloud+ 订阅的 iPhone 、iPad 或者 mac并且确认在【设置】-【 iCloud 分区】或者系统偏好设置中开启 Private Relay。

![](https://images.xiaozhuanlan.com/photo/2021/179baa33962f652dcea1684dc4e1a43d.png)

### 服务器准备工作
对于通过应用程序或者 Safari 访问服务器，服务器需要为 Private Relay 做一些准备工作。

#### Private Relay 生效时，如何确认用户地理位置

![](https://images.xiaozhuanlan.com/photo/2021/8c01cc0130bd2979415b5771cc941763.png)

首先，通过识别代理 IP 地址，服务器可以确认使用 Private Relay 访问的连接。通过上面的介绍可以得知，在 Private Relay 生效时，只有本地网络供应商和 Ingress Proxy 可以知道访问的 IP 地址。在 Private Relay 执行的流程中，Ingress Proxy 会将 IP 地址转化为其所属的大致范围传递给Egress Proxy 。Egress Proxy 会根据 Inpress Proxy 传递来的大致位置随机从与之相关的一组 IP 地址中选择一个 IP 地址，因此这个 IP 也是可以被同一区域内多用户共用。每一个 IP 地址都被映射到明确的城市或地区，因此使用恰当的地理 IP 映射数据库，服务器仍将有能力获取明确的位置信息。

![](https://images.xiaozhuanlan.com/photo/2021/41e21ec993e80b36cea438fc170f8c40.png)

与此同时，用户可以通过 **IP Address Location** 中的设置选项  `Use Country and Time Zone` ， 这样就会在更大的区域范围内随机选择 IP 地址。

其次，Private Relay 确保用户不能够使用系统来仿造来自不同区域。

基于以上两点，服务器可以继续设置基于地区的访问权限。

#### 使用 Private Relay 的网络连接

![](https://images.xiaozhuanlan.com/photo/2021/41318b862cd8bbae8f9f4871ace581bb.png)

1. 当设备将要访问服务器，首先它将建立一个与入口代理的连接（连接是使用网络供应者分配的 IP 地址建立的）；
2. 接下来设备使用入口代理将网络请求通过入口代理 IP 地址转发给出口代理；
3. 出口代理通过选择映射到设备所在城市或地区的 IP 地址，将这些请求转发到目标服务器；

通常，服务器和网站应该停止单独依赖客户端 IP 地址来判断用户地理位置或身份。如果需要访问地理位置，请考虑显式地请求需要精度的用户位置；如果你需要识别用户，请求用户登录或者其他形式的明确标识，而不是假定 IP 地址绑定标识。

## Private Relay 生效时管理网络的注意点
当使用 Private Relay 时，如果你正在本地网络上追踪包，将看到一些新的传输模式：

 - 你将看到更多的传输运行在 UDP 443 端口。这是用于与 Private Relay 代理通信的` QUIC（或HTTP/3）`传输。可以通过允许网络 UDP 443 端口以及确保路由器或网络地址转换器能够很好地处理，来确保网络传输工作良好。
 - 网络中会有更少的明文 UDP DNS 查询

下面是不使用 Private Relay 时，一个来自设备典型的网络连接请求
![](https://i.loli.net/2021/06/16/SPwaeHvQUc7Cf9g.png)
当设备尝试访问服务器，首先它发送了一个服务器主机名的 DNS 查询。一旦主机名被解析为一个 IP 地址，设备会使用传输协议（例如 TCP ）连接到服务器的 IP 地址。然后设备和服务器执行` TCP 三次握手`，进行 TLS 交换以建立和服务器的安全连接。
![](https://images.xiaozhuanlan.com/photo/2021/3e57e4d18539387b35207cb7007eddf8.png)
但是如你所见，如果没有 Private Relay ，通过简单地观察网络上的数据包，服务器的主机名和连接到服务器的设备 IP 地址都是可见的。

当使用 Private Relay
![](https://i.loli.net/2021/06/16/Pca6I7m5kqSEyij.png)

![](https://images.xiaozhuanlan.com/photo/2021/a30a7561de98860dbdad229323255648.png)

首先设备使用` QUIC `或者` HTTP/3 `建立与入口代理的连接。通过捕获的数据包，可以注意到发送到入口代理端口 443 的 UDP 数据包。一旦确立了到入口代理的网络连接，对服务器的访问就在入口代理的连接中得到了保护。对于服务端，协议没有任何变化。` TCP/TLS `交换与没有使用 Private Relay 时类似，唯一的区别是服务器是从代理 IP 地址而不是设备的 IP 地址接收传入的连接。

![](https://images.xiaozhuanlan.com/photo/2021/8f7a0f79702bcf4dde3df7af33c0f2ae.png)

大多数网络不需要审计或者监控所有用户传输。然而如果是运行在企业或者学校的网络，你的网络可能需要有拦截所有传输的策略。因此，你可以阻止 iCloud Private Relay 代理服务器的主机名。接下来当一台设备连接你的网络，用户将收到一个提示，表明 Private Relay 在当前网络上是被阻止的。用户可以选择是在这个网络禁用 Private Relay 还是切换到其他网络。对于家长控制，最好的解决方式是使用` NetworkExtension `框架提供的内容过滤 API。即使使用 Private Relay ，也会允许设备审核传输。

## 与 VPN 的对比
有些人会将 iCloud Private Relay 理解为苹果的“ VPN ”，这么理解还是有一些偏颇，相较于 VPN 两者还是有区别的。

1. 传统的 VPN 会在传输整个传输过程中隐藏 IP 地址，Private Relay 并非如此；
2. VPN 会将设备传输的所有信息加密，Private Relay 加密只会覆盖 Safari 、设备上与 DNS 相关、部分 APP 的传输；
3. VPN 可以越过地理位置封锁，但是 Private Relay 明确是遵守地理位置封锁的，没有隐藏设备所属地区或城市；
4. VPN 提供商本身知道你的真实IP和你正在访问的网站，这样对于 VPN 供应商的信任就成了新的问题。Private Relay 使用双跳架构：Ingress Proxy 会获取用户的 IP 地址，但不知道将要访问的服务名称；Egress Proxy 知道访问的服务名称，但不知道用户的IP地址。
