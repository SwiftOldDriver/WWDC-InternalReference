# WWDC21 10142 - 使用 HLS 让媒体无缝切换

![img](https://images.xiaozhuanlan.com/photo/2021/6683a812bb84bee9c7ecccb9c18b8efa.png)

# Overview

探索如何创建在剧集、歌曲、场景之间无缝转换的流媒体内容。 通过无缝 HLS 播放，您可以将多条内容动态地拼接在一起。您可以用来定制您的锻炼、设计交互式内容、讲述有趣的故事等等。 

我们将向您展示如何为流媒体音乐、事件的记录和预先录好的视频提供实实在在的连续性，并在您的 App 中提供引人入胜的观看体验。

# 本篇

[HTTP Live Streaming (HLS) ](https://developer.apple.com/documentation/http_live_streaming)是苹果公司在 2009 年推出的基于 HTTP 的流媒体协议。相较于实时传输协议 RTP，HLS 可以穿过任何允许 HTTP 数据通过的防火墙或者代理服务器，它也很容易使用 CDN 来传输媒体流，因而得到了广泛的应用。

本 session 介绍的是 HLS 无缝切换的新特性，由苹果流媒体团队的工程师 Simon Goldrei 讲解。

## 不够丝滑的 AVQueuePlayer

我们通常在需要播放一段视频时使用 AVPlayer。而 AVPlayer 一次只能播放一个视频文件。对于一次性播放多个视频的情况，苹果推出了 AVQueuePlayer 供开发者使用。

> AVQueuePlayer 是 AVPlayer 的一个子类，它在 AVPlayer 的基础上内部维护了一个播放列表，并对外提供了一系列 API 去播放这个列表。具体的 API 可以浏览[官方文档](https://developer.apple.com/documentation/avfoundation/avqueueplayer)。

虽然 AVQueuePlayer 提供了播放多个媒体的能力，但在媒体切换的过渡点上还存在一些瑕疵。特别是播放 HLS 媒体的时候，会存在声音毛刺、短暂的画面卡顿甚至是一个不知道何时才能完成的缓冲条等现象。

## 无缝切换带来的好处

而本篇 session 便来介绍如何解决这个问题，让媒体之间能够无瑕疵地进行切换，让用户体验丝滑般的享受。无缝切换能带来许多好处：

- 对于专辑类媒体能提供一个符合用户长久以来习惯的无缝体验
- 线性编码(这个没有具体说明是啥 >_<，笔者个人理解是能直接通过 AVQueuePlayer 设定播放顺序？)
- 提供一个动态的、交互式场景的体验。这里举个跑步健身的例子，你可以设定三个步骤，热身、有氧运动、冷却。对每个步骤分别设定一首 BGM，比如热身的时候设定「运动员进行曲」，运动时来首激情的摇滚，运动完听一曲经典萨克斯「Going Home」，回忆回忆当年夕阳西下踢完足球放学回家的滋味(误)。

## 无缝切换的条件

前面说的这么好。那么，如何才能让 AVQueuePlayer 支持无缝切换呢？是让用户对着屏幕喊一声“芝麻开门”吗？当然不是。要支持无缝切换，需要达到一定的条件才行。

![img](https://images.xiaozhuanlan.com/photo/2021/45835d9a3294c5d30c84a61e4ab3014d.png)

你的 HLS 播放列表需要满足以下几个条件：

- 需要使用苹果的 Fairplay Streaming 技术

> [FairPlay](https://developer.apple.com/streaming/fps/) 是苹果公司研发的一种数字版权管理 (DRM) 技术。它将加密的 AAC 音频层内置于 MP4 多媒体文件格式中，以保护通过 iTunes Store 发售作品的版权，只允许经过授权的设备才能播放内容。

> 关于 Fairplay Streaming 技术可以观看 [WWDC20-10655](https://developer.apple.com/videos/play/wwdc2020/10655/) 了解详尽信息。

- 需要在 `EXT-X-STREAM-INF` 标签的 `CODECS` 属性中指定编解码器
- 需要在 `EXT-X-MEDIA` 标签的 `CHANNELS` 属性中指定频道数量
- 变体之间的配置需要保持一致。比如两段音频之间的采样率(Sample Rate)和位深(Bits Depth)需要保持一致，否则无缝切换会失效
- 需要遵守 CMAF 标准，同时 Simon 还建议使用 elst(Edit List Box) 来表示起始帧和剩余帧的存在

> [CMAF(Common Media Application Format)](https://developer.apple.com/documentation/http_live_streaming/about_the_common_media_application_format_with_http_live_streaming)是一个可扩展的标准，用于分段媒体的编码和打包，以便自适应地分发到用户终端设备上进行解码。CMAF 抽象出了一个应用模型，该模型存在多个实现，包括 HLS 和 MPEG DASH。
>
> elst 全称 Edit List Box，mp4 文件中不一定都含有这个 box。该 box 作用是使某个 track 的时间戳产生偏移。苹果在 iOS 13.1 以后支持使用 elst，详情见[官方文档](https://developer.apple.com/documentation/http_live_streaming/preparing_audio_for_http_live_streaming?language=objc)。关于 elst 的内容可以参考[mp4文件elst研究](https://blog.jianchihu.net/mp4-elst-box.html)。

内容提供方在遵守了以上的条件之后就能进行无缝的切换了。下面举个例子。

> 注意，这个例子中包含媒体组(Media Group)相关技术，细节请参考[官方文档](https://developer.apple.com/documentation/http_live_streaming/example_playlists_for_http_live_streaming/adding_alternate_media_to_a_playlist?language=objc)。

```
#EXTM3U

#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="aac-lc",CHANNELS="2",URI="./AAC_LC/prog_index.m3u8"
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="he-aac",CHANNELS="2",URI="./HE_AAC/prog_index.m3u8"

#EXT-X_STREAM-INF:BANDWIDTH=3900000,CODECS="avc1.64001f,mp4a.40.5",AUDIO="he-aac"
./720P/prog_index.m3u8

#EXT-X_STREAM-INF:BANDWIDTH=4000000,CODECS="avc1.64001f,mp4a.40.2",AUDIO="aac-lc"
./720P/prog_index.m3u8

#EXT-X_STREAM-INF:BANDWIDTH=8900000,CODECS="avc1.640028,mp4a.40.5",AUDIO="he-aac"
./1080P/prog_index.m3u8

#EXT-X_STREAM-INF:BANDWIDTH=9000000,CODECS="avc1.640028,mp4a.40.2",AUDIO="aac-lc"
./1080P/prog_index.m3u8
```

这个播放列表中有四个变体。其中一对是 720p，另外一对是 1080p。每一对都分别设定两个不同的编解码器，定义为 he-aac 和 aac-lc。假设当前在 he-aac 和 720p 条件下播放，那么播放下一段媒体时，AVQueuePlayer 将会选择同样是 he-aac 和 720p 的变体来播放。

这个选择不会因为网络的好坏而受到影响。即便在播放下一段前一刻网络速度爆表，也不会切换到更高清的变体上去。当然，如果切换完成后网络条件还是很好，那么此时播放器会切换到更高清的变体上去。

那么如果下一个 m3u8 主播放列表中没有提供 he-aac 的变体怎么办？

```
#EXTM3U

#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="aac-lc",CHANNELS="2",URI="./AAC_LC/prog_index.m3u8"
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="he-aac",CHANNELS="2",URI="./HE_AAC/prog_index.m3u8"

#EXT-X_STREAM-INF:BANDWIDTH=4000000,CODECS="avc1.64001f,mp4a.40.2",AUDIO="aac-lc"
./720P/prog_index.m3u8

#EXT-X_STREAM-INF:BANDWIDTH=9000000,CODECS="avc1.640028,mp4a.40.2",AUDIO="aac-lc"
./1080P/prog_index.m3u8
```

播放器无法找到一个条件一致的变体，无缝切换也就失效了。此时播放器会自己选择一个合适的变体进行切换。

## 代码示例

Simon 提供了一个 AVQueuePlayer 的简单示例。

```swift
let item1 = AVPlayerItem(url: url1)
let item2 = AVPlayerItem(url: url2)

let player = AVQueuePlayer()

player.insert(item1, after: nil)
// 设置第二段视频紧接着第一段播放
player.insert(item2, after: item1)

player.play()
```

## 小技巧——资源切割

此外，Simon 还分享了一个资源切割的技巧。简单地说，就是可以在同一资源上划分多个片段进行播放。在代码中就是在一个 AVAsset 中实例化出多个 AVPlayerItem 播放。

这么做的好处是可以动态地选择需要播放的片段和调整播放顺序，相当于可以使用代码重新剪辑资源。

实现这个功能需要依赖 AVPlayerItem 的 `seekToTime` 和 `forwardPlaybackEndTime` 这两个方法。使用 `seekToTime` 设置起始点，`forwardPlaybackEndTime`设置终止点。再通过 AVQueuePlayer 将分割的片段连起来播放。

![img](https://images.xiaozhuanlan.com/photo/2021/8019b31dc4993cff65c4e6e2bac48721.png)

代码示例：

```swift
let asset = AVAsset(url: url)

let startTimes: [CMTime] = [startTime1, startTime2, startTime3]
let endTimes: [CMTime] = [endTime1, endTime2, endTime3]

let item0 = AVPlayerItem(asset: asset)
item0.seek(to: startTimes[0], completionHandler: nil)
item0.forwardPlaybackEndTime = endTimes[0]

let item1 = AVPlayerItem(asset: asset)
item1.seek(to: startTimes[1], completionHandler: nil)
item1.forwardPlaybackEndTime = endTimes[1]

let item2 = AVPlayerItem(asset: asset)
item2.seek(to: startTimes[2], completionHandler: nil)
item2.forwardPlaybackEndTime = endTimes[2]

let player = AVQueuePlayer()

// 此处改变了原本的播放顺序
player.insert(item1, after: nil)
player.insert(item2, after: item1)
player.insert(item0, after: item2)

player.play()
```

# 结语

使用无缝切换技术可以让我们为用户提供一个流畅的音视频体验。无缝切换技术本质上还是减少或避免因参数变化而导致的切换性能损耗。它可能不是众多新功能中改动最大的，但有一点可以肯定的是，无缝切换是最能体现苹果极致的工匠精神的特性之一。
