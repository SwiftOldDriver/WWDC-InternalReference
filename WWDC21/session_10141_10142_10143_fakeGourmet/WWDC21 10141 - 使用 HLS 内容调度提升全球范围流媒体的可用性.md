## Overview

本文基于 [Session 10141](https://developer.apple.com/videos/play/wwdc2021/10141) 梳理。

在本文中我们将从实际问题出发，由浅入深地探索如何利用 HLS 内容调度（Content Steering）功能动态更新 CDN 冗余策略，从而实现负载均衡、故障转移等。

## 本篇

如今的网络连接是全球范围的，流媒体提供商在世界各地提供他们的内容服务。然而，在全球范围内分发内容存在着大量的挑战，这些挑战行业必须解决。Apple 的 HLS 流媒体团队也认识到这些具有挑战性的问题，并不断努力提供解决方案。

> [HTTP Live Streaming (HLS) ](https://developer.apple.com/documentation/http_live_streaming)是苹果公司在 2009 年推出的基于 HTTP 的流媒体协议。因为 HLS 是基于 HTTP 协议的流媒体传输协议，所以 HLS 可以复用任何 HTTP 相关的基础设施。相对于 RTP 来说，HLS 被防火墙拦截的概率会更小（因为大部分防火墙都允许 HTTP 数据通过），也更容易使用 CDN 来传输媒体流，因此 HLS 得到了广泛的应用。

本 session 介绍的是 HLS 的内容调度（Content Steering）功能，由苹果 HLS 流媒体团队的工程师 Zheng Naiwei 讲解。

> 注意，在阅读本文前我们假设你已经对 HLS 技术有所了解。如果你还不是很了解，建议先阅读 [iOS 边学边记 HLS 协议 m3u8 ts 详解](https://blog.csdn.net/u014219492/article/details/110184490)。

如果你想了解其他有趣的 HLS 新特性，请参考：

## 内容调度能做什么？

内容调度聚焦于提升流媒体的可用性，具体来说包括解决网络拥塞缓解和错误恢复等问题。

为了便于理解内容调度的作用，举两个例子。

### 网络拥塞缓解

假设亚洲一家流行的流媒体提供商在以下三个国家部署了 CDN 基础设施：中国、日本和新加坡。

在下面这张图中，一切都在完美状态下工作。我们有来自中国 CDN 的中国用户和来自日本 CDN 的日本用户，这两个 CDN 网络都很流畅。

![](https://images.xiaozhuanlan.com/photo/2021/18385eb5e0d474ade6671a66c092bd8c.png)

现在我们引入一些变化。假设来自中国 CDN 的用户的数据流量猛增，达到让中国 CDN 服务过载的程度，此时中国区发生网络堵塞。以现有的 HLS 技术，很难去解决这个网络堵塞问题。即便可以让堵塞之后加入网络的用户切换至一个不同区的 CDN 服务器，但现存的用户没法切换，网络拥堵会持续地让 CDN 过载。

![](https://images.xiaozhuanlan.com/photo/2021/e5240cf61262ab0776c2076a908a3c8e.png)

借助 HLS 内容调度功能，内容提供商可以通过托管自己的调度服务器来解决此问题。它与所有最新的 HLS 客户端建立通道，更新最新的 CDN 策略。例如，它可以向中国 30% 的现有客户发送切换到日本 CDN 的更新策略。接收更新策略的客户端会将其网络流量重定向到日本 CDN。

![](https://images.xiaozhuanlan.com/photo/2021/c27fd0edd3b2c0727f40bc1075bda476.png)

由于调度服务器的介入，中国 30% 现存的用户收到 CDN 更新策略后成功切换到日本 CDN，因此中国 CDN 的网络拥塞问题得到了缓解。

![](https://images.xiaozhuanlan.com/photo/2021/695763880a9ea1f79ab6f5e00563a277.png)

### 错误恢复

网络拥塞是众多可用性问题中的一种。我们再举一个网络中断的例子。

假设从日本用户到日本 CDN 的网络路径发生了区域中断。

使用我们已有的 HLS 技术，内容提供商可以在主播放列表中列出我们的备用变体（variant）流，因此客户可以尝试遍历列表，直到找到一个有效的 CDN。

但这种方式是不够完善的，因为内容提供商无法实时更改主播放列表中列出的 CDN 的顺序或优先级。

![](https://images.xiaozhuanlan.com/photo/2021/e09a154f81a002d95edc03c7a00c5c71.png)

为了完整解决区域中断问题，内容调度功能“应运而生”。在调度服务器（Steering Server）的帮助下，客户端可以近乎实时地更新最新的 CDN 优先级。

在网络中断之前，日本的客户端会定期从调度服务器获取 CDN 优先级列表更新。

![](https://images.xiaozhuanlan.com/photo/2021/94a18f184fcf041bb60281aa7bbda55d.png)

假设在最近一次更新中，日本用户收到以下一个 CDN 优先级列表，列表中日本 CDN 的优先级最高，其次是新加坡，最后是中国。在此之前，用户在日本 CDN 上接收和传输流媒体。

当发生网络中断时，日本 CDN 将从优先级列表中标记为不可用。

于是客户端将移至优先级列表中的下一个 CDN（在本例中既为新加坡的 CDN），并重定向其网络流量。

![](https://images.xiaozhuanlan.com/photo/2021/ee4afac725cd61887e2fc6a8542c0436.png)

重定向至新加坡网络后，日本用户就能够流畅地观看他们的节目了。

## 如何集成内容调度功能？

以上示例展示的是内容调度功能的高级概述。现在让我们更深入地看看如何将这个功能集成到流媒体中。

### 不支持内容调度的 m3u8

下面一个例子是一个简单 m3u8 列表，它单纯复制了所有变体的不同 CDN URL。如前所述，这个播放列表的 CDN 排序是写死的，无法在播放过程中更改它。

```
#EXTM3U

#EXT-X-STREAM-INF :BANDWIDTH=1280000, CODECS="avc1.640028"
https://cn.example.com/low/audio-video.m3u8
#EXT-X-STREAM-INF :BANDWIDTH=7680000, CODECS-"avc1.640028"
https://cn.example.com/hi/audio-video.m3u8

#EXT-X-STREAM-INF :BANDWIDTH=1280000, CODECS="avc1.640028"
https://jp.example.com/low/audio-video.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=7680000,CODECS="avc1.640028"
https://jp.example.com/hi/audio-video.m3u8

#EXT-X-STREAM-INF:BANDWIDTH=1280000,CODECS="avc1.640028"
https://sg.example.com/low/audio-video.m3u8
#EXT-X-STREAM-INF :BANDWIDTH=7680000, CODECS="avc1.640028"
https://sg.example.com/hi/audio-video.m3u8
```

### 支持内容调度的 m3u8

现在让我们看看如何让这个播放列表支持内容调度。主要额外增加了一个标签和两个关键字段:

- CONTENT-STEERING 标签
- PATHWAY-ID 字段
- SERVER-URI 字段

这个改动很小，同时也是向后兼容的。让我们来看看添加后的内容。

```
#EXTM3U
#EXT-X-CONTENT-STEERING:SERVER-URI="/steering?video=00012",PATHWAY-ID="CN"

#EXT-X-STREAM-INF :BANDWIDTH=1280000, CODECS="avc1.640028",PATHWAY-ID="CN"
https://cn.example.com/low/audio-video.m3u8
#EXT-X-STREAM-INF :BANDWIDTH=7680000, CODECS-"avc1.640028",PATHWAY-ID="CN"
https://cn.example.com/hi/audio-video.m3u8

#EXT-X-STREAM-INF :BANDWIDTH=1280000, CODECS="avc1.640028",PATHWAY-ID="JP"
https://jp.example.com/low/audio-video.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=7680000,CODECS="avc1.640028",PATHWAY-ID="JP"
https://jp.example.com/hi/audio-video.m3u8

#EXT-X-STREAM-INF:BANDWIDTH=1280000,CODECS="avc1.640028",PATHWAY-ID="SG"
https://sg.example.com/low/audio-video.m3u8
#EXT-X-STREAM-INF :BANDWIDTH=7680000, CODECS="avc1.640028",PATHWAY-ID="SG"
https://sg.example.com/hi/audio-video.m3u8
```

首先最值得注意的是，我们现在通过 PATHWAY-ID 字段将变体流分组到不同的路径中，每个路径通常对应一个 CDN。

在上面这个例子中，我们将变体流分为 CN、JP 和 SG 三个路径，分别代表中国、日本和新加坡的 CDN。

在复杂业务中，路径还可能会包含来自不同 CDN 的变体流。当客户端指定了某一路径时，就只能选择来自该路径的变体流。

新加的 CONTENT-STEERING 标签声明需要支持内容调度，标签上的 PATHWAY-ID 字段指定了在播放启动时使用的初始路径。在这种情况下，客户端在开始播放时只从 CN 路径的变体流中进行选择。

而另外一个 SERVER-URI 字段会指定调度服务器的位置。客户端将从该 URI 中获取更新策略。

### 让内容调度支持媒体组

如果您的播放列表包含媒体组，集成内容调度也很简单。

> 媒体组（Media Group）为 HLS 提供了一种外挂音频、视频、字幕的格式，可以在不改动已生成的 HLS 分片信息的情况下，为客户端提供新的可选媒体信息，通常在多语言场景下使用。详情请参考 [官方文档](https://developer.apple.com/documentation/http_live_streaming/example_playlists_for_http_live_streaming/adding_alternate_media_to_a_playlist?language=objc)。

所要额外做的就是为每个路径分配一个唯一的 GROUP-ID。下面的播放列表中每个路径都对应了一个音频组，比如将 GROUP-ID 设置为 CN 路径的 CN-audio。下面是一个示例。

```
#EXTM3U
#EXT-X-CONTENT-STEERING:SERVER-URI="/steering?video=00012",PATHWAY-ID="CN"

#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="CN-audio",NAME="English",URI="https://cn.example.com/audio.m3u8"
#EXT-X-STREAM-INF :BANDWIDTH=7680000,CODECS-"avc1.640028",AUDIO="CN-audio",PATHWAY-ID="CN"
https://cn.example.com/hi/audio-video.m3u8

#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="JP-audio",NAME="English",URI="https://jp.example.com/audio.m3u8"
#EXT-X-STREAM-INF:BANDWIDTH=7680000,CODECS="avc1.640028",AUDIO="JP-audio",PATHWAY-ID="JP"
https://jp.example.com/hi/audio-video.m3u8

#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="SG-audio",NAME="English",URI="https://sg.example.com/audio.m3u8"
#EXT-X-STREAM-INF :BANDWIDTH=7680000,CODECS="avc1.640028",AUDIO="SG-audio",PATHWAY-ID="SG"
https://sg.example.com/hi/audio-video.m3u8
```

## 内容调度如何工作？

现在我们的 m3u8 列表已经集成了内容调度了，让我们来看看内容调度的技术细节。首先是启动流程。

### 内容调度启动流程

当 HLS 客户端加载带有内容调度标签的主播放列表时，启动流程会有所不同。

1. 起始时只会使用来自初始路径的变体流。
2. 加载并开始播放所选的变体流及其媒体片段，这与没加内容调度的流程一致。
3. 在播放的同时客户端会在后台并行地定时发出调度清单请求。

### 调度清单

让我们来看看调度清单（Steering Manifest）请求是什么样的。这是一个简单的 HTTP GET 请求，它的构成结构是

```
主播放列表 URL + SERVER-URI 字段内容
```

我们举个例子来剖析它的格式。

比如我们 m3u8 主播放列的 URL 为`https://example.com`，同时我们 SERVER-URI 字段的内容为`/steering?video=00012`，那么我们最终的请求为 `https://example.com/steering?video=00012`。

![](https://images.xiaozhuanlan.com/photo/2021/3941341e3f78550e8efd71df772d8f9c.png)

当客户端向调度服务器发送完请求后，调度服务器会返回给该客户端一个调度清单。其表现形式为 JSON。下面是个调度清单的例子。

```json
{
    "VERSION": 1,
    "TTL": 300,
    "PATHWAY-PRIORITY": ["CN", "JP", "SG"]
}
```

如上所示，调度清单共有三个字段。

VERSION 字段用于区分更新版本。TTL 字段用于设定请求时间间隔。而最重要的属性是 PATHWAY-PRIORITY 字段。这是一个按优先级排序的路径 ID 列表，排第一个的是最优先的。

在此示例中，我们优先选择 CN 路径，其次是 JP，最后是 SG。

调度服务器可以为每个客户端定制调度清单，这使调度服务器在托管网络流量方面具有巨大潜力。

### 内容调度工作流程

客户端在收到来自调度服务器的调度清单后，首先会进行内容调度评估，来确定是否切换到不同的路径。

首先，客户端将排除不符合选择条件的变体流，包括：

- 由于网络错误而被标记不可用的变体流。
- 带有未支持的编解码器的变体流。

在排除了不符合条件的变体流后，客户端就会从余下的变体流中选择优先级最高的那个。

如果选择的路径与当前使用的路径不同，客户端将立即切换到新路径。

如果没有选择合适的路径或已经在使用选择的路径，客户端将什么都不做，继续使用当前路径进行播放。

然后，客户端会发送下一个调度清单请求，从调度服务器中获取更新。

之前也提到，有一个 TTL 字段，用于指定客户端在发送下一个调度清单请求之前应等待的秒数。在此示例中，客户端会使用调度清单中的路径优先级列表执行内容调度评估，然后它将在 300 秒（即 5 分钟）内发送下一个调度清单请求。

另外，调度服务器能够在每个调度清单响应中更改此 TTL 值，从而影响客户端应安排下一个请求的时间。比如你可以让调度服务器向每个客户端的 TTL 值设置一些随机数来分配负载，这很有用。

此外，还有个可选的 RELOAD-URI 字段，它告诉客户端在何处请求下一个调度清单。调度服务器可以利用此字段在 URL 中注入客户端状态或上下文，以便在下一个请求中使用。

下面是一个带有 RELOAD-URI 字段的示例。

```json
{
    "VERSION": 1,
    "TTL": 300,
    "RELOAD-URI": "https://example.com/steering?video=00012&session=123",
    "PATHWAY-PRIORITY": ["CN", "JP", "SG"]
}
```

调度服务器在这个 RELOAD-URI 字段中注入了 session=123 的上下文。

## 重识内容调度

现在我们已经了解了内容调度的主要技术细节。让我们回到我们早期的示例之一，看看它是如何工作的。

假设在这种情况下，中国的所有用户都收到了一个以 CN 路径为首选的调度清单，并且由于用户数量的增加，对应的中国 CDN 正在处于过载状态。

于是调度服务器开始工作，向中国 30% 的用户发送一个不同的调度清单，将偏好的路径改为 JP。

![](https://images.xiaozhuanlan.com/photo/2021/a47dc06f1d9f32a561284cf5a361baa1.png)

在进行内容调度评估后，发现此时 JP 路径的优先级最高，因此这 30% 的中国客户将切换到 JP 路径，将他们的网络流量重定向到日本的 CDN，从而解决中国 CDN 的网络拥塞情况。

![](https://images.xiaozhuanlan.com/photo/2021/d2da1f51d0d22cb80f45f66ddfda5ca5.png)

## 结语

内容调度的引入很大提升了 HLS 的可用性，使用内容调度技术可以为我们解决以前无法着手的网络拥塞和网络错误等问题。对于如今全球化的流媒体环境，一个高可用的流媒体服务是内容提供商从众多竞争者中脱颖而出的优势之一。如果你的用户还在反馈你的内容服务网络不够流畅，那不妨试试接入内容调度这个新功能吧。
