## 概述
近几年，Apple 加强了对于 iPad 具有强大生产力的宣传，并且推出了 Magic Keyboard 等体验并不怎么样的键盘（给你提供了好功能却让你用的不怎么舒服的 Apple 才是真 Apple）。在此环境下，人们越来越喜欢在 iPad 上搭配硬件键盘使用，那么给你的 iPadOS App 添加键盘导航功能将会更加友好 👬 。Mac Catalyst 和 iPadOS 15 引入了键盘导航功能，使用 Tab 键在应用程序的重要区域之间导航，使用箭头键在一个区域内导航。通过本文，你将了解到如何使应用中的更多内容可聚焦，以及如何自定义这些可聚焦项目的外观。我们还会讨论到一种在应用中定义画面结构的新方法以及响应者链中的一些变化。

## 添加 Keyboard navigation

想要获得键盘导航功能十分简单，使用 iOS 15 SDK 编译后，无需适配便能获得使用 Tab 键在文本字段 (UITextFiled)、文本视图（UITextView）和侧边栏之间导航的功能。除此之外，我们还应当关注文本输入、列表和集合视图，使用户能够通过键盘导航使用应用程序的关键功能。对于 Controls ，例如 UIButton、UISegmentControl、UISwitch，我们可以在 [Support Full Keyboard Access in your iOS app](https://developer.apple.com/videos/play/wwdc2021/10120/) 找到处理的方式，无需单独为其添加键盘导航功能。
iPadOS 上的键盘导航使用与之前的 tvOS 应用程序具有相同的 Focus System。如果你曾使用过，那么许多 API 都会十分熟悉，仅需注意新的 API 和某些行为差异。

## 使更多的 UI 元素可聚焦

现在，我们来看看如何让更多的 UI 元素可聚焦。`canBecomeFocused` 是唯一能够让 UIView 获得键盘导航功能的属性，它是 `UIFocusItem` 的只读属性。重写此属性并返回 true 使项目可被聚焦，此时可以将此项目称为可聚焦项。

```swift
override var canBecomeFocused: Bool { true }
```

那么，什么是可聚焦项? 首先来理解 Focus System，在 Focus System 中主要有两个协议：`UIFocusItem` 和 `UIFocusEnvirment`。FocusItem 就是可聚焦的项目，FocusEnvironments 定义可聚焦项的层次结构。UIView 符合这两个协议，因此任何视图都可以被聚焦，同时可以包含其他可被聚焦的子视图。因为 UIViewController 不提供任何视觉效果，它只遵循了 UIFocusEnvironment 协议。

![](https://images.xiaozhuanlan.com/photo/2021/f2d378d1e3a7988617cbfc3cd1b75517.png)

我们还可以在自己的对象上实现两个协议，从而实现使用其他技术（例如 Metal）呈现的内容中采用 Focus System。
通常，我们需要将 table 或者 collection 视图的 cell 作为导航的候选对象。UIKit 为我们提供了一些方便的 API，无需子类化 Cell。仅需在  table 或者 collection 视图上设置 `allowedFocus` 为 true 使所有的 Cell 都可以被聚焦。

```swift
// 设置所有的 cells 可聚焦
class MyViewController: UICollectionViewController {
    override func viewDidLoad() {
        super.viewDidLoad()
        self.collectionView.allowsFocus = true
    }
}
```

使用 `canFocusItemAt indexPath` 方法，我们也可以在代理中实现更细粒度的控制。

```swift
// 单独设置某个 cell 可聚焦
class MyCollectionViewDelegate: NSObject, UICollectionViewDelegate {
    func collectionView(_ collectionView: UICollectionView,
                canFocusItemAt indexPath: IndexPath) -> Bool {
        // some logic in your app
        return true
    }
}
```

以上两种设置方法都不会对重写了canBecomeFocused 的 Cell 产生影响。

## 自定义可聚焦项外观

在整个 Focus system 中， 我们通常会看到两种样式，光晕效果以及单元格的高亮效果。

### 光晕效果

光晕效果是单个可聚焦项目被选中时的默认效果，这是 Mac Catalyst 上的默认效果，类似于 macOS 中 focus ring。在 iPadOS 上，我们可以通过将 UIFocusHaloEffect 分配给 focusEffect 属性来使用此效果。直接初始化 UIFocusHaloEffect，系统会为我们推断 effect 的形状，更好的方法是自定义形状以更好匹配内容。例如，要匹配一个圆角的、带有 badge 的图片，我们需要添加以下代码：

```swift
// 初始化一个 Halo Effect 并使其外观与 Cell 相适应
let focusEffect = UIFocusHaloEffect(roundedRect: self.bounds, cornerRadius: self.layer.cornerRadius, curve: .continuous)

// 设置 focusEffect 在视图层次结构中的相对顺序
focusEffect.referenceView = self.imageView

// 设置 focusEffect 的 super view
focusEffect.containerView = self.containView

self.focusEffect = focusEffect
```
上面的代码初始化一个和可聚焦项目bounds cornerRadius 相匹配的 halo effect，设置 referenceView 来控制 effect 在图层间的相对位置，设置 containerView 以更好匹配内容。
![优化图层顺序前后对比](https://images.xiaozhuanlan.com/photo/2021/d4c8dd1514c1346216161236c9de0aaf.png)

### 单元格的高亮效果（Cell highlighting）

集合和表格视图中的单元格只有在它们具有完全不透明的内容（如图像）时才应该在它们周围显示光环，为了获得更好的效果，建议为单元格设置背景颜色。在所有其他情况下，当一个单元格成为焦点时，它应该看起来是高亮的。这意味着背景应该变成 tinyColor，并且应该调整文本和图标的前景色以获得良好的对比度。此高亮外观不可用作 UIFocusEffect。
如果使用 iOS 14 中引入的背景和内容配置，我们将自动获得这种外观（更多关于 [Modern cell configuration](https://developer.apple.com/videos/play/wwdc2020/10027/)）([《WWDC20 10027 - 现代cell配置》](https://xiaozhuanlan.com/topic/6923780145))。不使用背景和内容配置的情况，我们可以通过下面示例代码来了解如何在所有情况下获得正确的颜色。
首先，需要确保的是：想要应用自定义的外观，最好将 focusEffect 属性设置为 nil，这会关闭系统样式避免影响。

```swift
init(frame: CGRect) {
   super.init(frame: frame)
   self.focusEffect = nil
}
```

接下来，我们在需要配置外观的 Cell 中重写 `didUpdateFocus (in: context withAnimationCoordinator :)` 方法。如果下一个被聚焦的项目是 self，那么应用定义的外观来指示焦点。如果先前聚焦的项目是 self，则恢复未聚焦时的外观。

```swift
func didUpdateFocus(in context: UIFocusUpdateContext, withAnimationCoordinator coordinator: UIFocusAnimationCoordinator) {
    if context.nextFocusedItem == self {
        // This view is focused. Customize its appearance.
    }
    else if context.previouslyFocusedItem == self {
        // This view was focused.
    }
}
```

当下一个或者上一个可聚焦项目与当前项目相关时，我们应当在 `didUpdateFocus (in: context withAnimationCoordinator :)` 中进行更改，而非其他地方。这是因为先前聚焦项的所有祖先环境以及下一个聚焦项都会收到对 `didUpdateFocus(in: context)` 的调用，所以每个父级和其所属 view controller 都会得到这个调用。这允许非常灵活的实现，其中父级可以对子级的焦点变化做出反应。

![didUpdateFocus 的调用](https://images.xiaozhuanlan.com/photo/2021/328f68f83338ab5c6f141f65150310e8.png)

**Selection 不同于 Focus**

Table 和 collection 视图通常具有选择（选中）的概念，不过选择（Selection）和聚焦（Focus）是两个不同的概念，一般情况选择指用户选中了这个项目，而聚焦则可能不需要被选中。不过在侧边栏和类似的上下文中，当移动焦点时，选择通常是跟随焦点的。我们可以通过设置 table 或者 collection 视图的 `selectionFollowsFocus` 属性来控制大多数 Cell 需要的行为。如果需要单独控制某个 Cell 的行为，可以在代理中实现 `selectionFollowsFocusForItemAt indexPath` 。需要注意的是，设置 `selectionFollowsFocus` 为 false 在选择单元格导致破坏性操作发生时很有用，例如在点击单元格弹出新的 VeiwController 或 Alert。
例如，我们在在“照片”中，选择“新相册”会显示询问相册名称的提醒。

![破坏性操作取消聚焦](https://images.xiaozhuanlan.com/photo/2021/a07f19fd362ecc0001ebf619f7eb0a7c.png)

使用代理时，selectionFollowsFocus 属性的值仍然很重要。将 selectionFollowsFocus 设置为集合视图的整体意图，然后使用代理来表达单个单元格的特殊行为。在选择正确的行为时，系统将考虑这两个值。使用代理控制选择跟随焦点：

```swift
// customizable per cell
func collectionView(_ collectionView: UICollectionView, selectionFollowsFocusForItemAt indexPath: IndexPath) -> Bool {
    return self.action(for: indexPath).type != .showAlert
}

```

## 新的定义画面结构的方式 --- Focus Group

Focus Group 是一种用于键盘导航的新功能，也可以用于在应用程序中表达视图结构。UIKit 能够自动从层次结构中推断 focus group，我们也可以显示的声明这些层次结构以控制 Tab 键在我们的应用程序中移动焦点的方式。
在 iPadOS 中，定义了两种不同的键盘导航的方式：箭头键和 Tab 键。箭头键只能在应用程序的定义区域（Focus Group）内移动焦点；Tab 键则是用于焦点在 Focus Group 之间移动。在下图中，定义了提醒、搜索字段和列表三个区域，Tab 键可以让焦点区域在这三个区域之间移动，使用箭头键能够选中焦点区域中某个可聚焦项目 。

![](https://images.xiaozhuanlan.com/photo/2021/b4d4cb464aa160a61b1771eae2077a24.png)

当焦点移动到某个 Group 时，它会自动的聚焦在某个项目上，这个项目称为主要项目并且可更改。比如这里我们看到焦点正聚焦于第二个提醒事项。当我们使用 Tab 键离开这个 Group 时 Focus system 会记住这一点，当回到提醒列表时，焦点将再次移动到第二个提醒事项，因为这个项目现在是该组的主要项目。Tab 键连接每个 Group 的主要项目并在它们之间移动焦点，这称为选项卡循环。

![](https://images.xiaozhuanlan.com/photo/2021/c95e8638b0c03920c14cb53644f18e9a.png)

滚动视图，如集合和表格视图，以及文本输入类，如文本字段和文本视图，这些 Focus Envirment 拥有自己默认的 Group。如果 Focus Envirment 没有定义自己的Group ，它会继承父级 Envirment Group，通常是它的父视图或 ViewController。我们要使用自定义的 Focus Group，需要将 focusGroupIdentifier 分配给任意 View 或者 ViewController，两个 Envirment 显示或继承相同的 focusGroupIdentifier 时，它们则属于同一个 Group。

```swift
self.focusGroupIdentifier = "com.myapp.groups.sidebar"
```

要定义 Focus Group 的主要项目，则只需要配置 Group 中可聚焦项目的优先级，这定义了项目在组中的重要性，具有最高优先级的可见项是组的主要项目。ignored 是系统默认的优先级，同时自定义的优先级不应低于默认优先级。事实上，我们应该提高不同项目的优先级。

```swift
extension UIFocusGroupPriority {
    public static let ignored: UIFocusGroupPriority // 0
    public static let previouslyFocused: UIFocusGroupPriority // 1000
    public static let prioritized: UIFocusGroupPriority // 2000
    public static let currentlyFocused: UIFocusGroupPriority // NSIntegerMax
}
```

例如，在下面这段代码中，我们给某个单元格分配高于 `.previouslyFocused` 的优先级，则该单元格将变得比该组的最后一个被聚焦项目更重要。

```swift
// Customizing an item’s focus group priority

func collectionView(_ collectionView: UICollectionView, cellForItemAt indexPath: IndexPath) -> UICollectionViewCell {
    let cell = ...
    if self.isCallToActionCell(at: indexPath) {
        // This cell is not as important as a selected cell but should
        // be chosen over the last focused cell in this group.
        cell.focusGroupPriority = .previouslyFocused + 10
    }
    return cell
}
```

之前我们提到，连续使用 Tab 键会焦点从搜索区域转到侧边栏中的列表，然后转到右侧的提醒。这个顺序是 UIKit 默认完成的，所有的 Group 按照从左往右、从上往下的顺序切换。但是，如果我们自定义了类似的视图，比如侧边栏，Focus System 将不知道侧边栏是一个不同的列。此时，我们需要把这些自定义的视图放到一个公共组中，通过定义其  `focusGroupIdentifier`。

![Focus Group 的顺序](https://images.xiaozhuanlan.com/photo/2021/49b001758e32ba03971468996bc1514e.png)

通常情况下，我们不需要特地定义聚焦区域的选项卡顺序，Focus System 使用 Focus Group 结合阅读方向、布局和可见性等推导选项卡循环顺序，以提供一致的体验。
更多有关 Group 的实现，可以参考这个示例 [Navigating an App’s User Interface Using a Keyboard](https://developer.apple.com/documentation/uikit/uikeycommand/navigating_an_app_s_user_interface_using_a_keyboard)

## 响应链中的变化

**注意 becomeFirstResponder 的调用**

如果我们的应用程序有着对响应链的处理，那么注意在何处调用  becomeFirstResponder 是十分重要的。由于响应链和焦点是同步的，更改第一响应者将强制更新焦点，这可能会对应用程序造成极大的破坏，通常应当避免调用 becomeFirstResponder。

**自定义 KeyCommand 的处理**

程序中使用了 Tab 或向下箭头等自定义按键命令，iOS 15 SDK 编译后将不再起作用。不过如果此键盘命令之前用于构建自定义的键盘导航，我们可以保持不变，否则我们应当重新映射此键盘命令。如果确实想继续使用此键盘命令，请确保它不会中断键盘导航，然后将 WantsPriorityOverSystemBehavior 设置为 true。

```swift
keyCommand.wantsPriorityOverSystemBehavior = true
```

## SwiftUI 中的 Focus System

开箱即用是 SwiftUI 的一大优点，内置组件在各平台上提供了合理且熟悉的整体体验，这一特性同样体现在 SwiftUI 的 Focus System 中。
Focus System 可让我们的应用程序从键盘、遥控器、游戏控制器、可访问的开关控件和其他非触摸来源获取输入。大多数情况下，SwiftUI 会代替我们管理 Focus System。当用户点击文本字段或按下 Tab 键时，或者当用户点击可调节的手表复杂功能或在 Siri Remote 上滑动时，SwiftUI 会决定焦点如何受到影响以及如何指示其下一个位置。在今年的 WWDC 上，Apple 推出了新的 API 来加强这种体验。
我们通过一个登录的例子来看看这个新的 API。这个例子中，我们会显示一个电子邮件字段、一个密码字段和一个“使用 Apple 登录”按钮，以便用户可以登录帐户。在此画面上，当提交时检查邮箱格式不正确，我们希望把焦点聚焦在邮箱输入框内。以前的 SwiftUI，我们可能需要去写一些额外代码把 UIKit 中的 becomeFirstResponser 给移植过来；现在，为了实现这种行为，我们将使用 FocusState API。

![](https://images.xiaozhuanlan.com/photo/2021/989642afe332229b77c59bf8e2096cba.png)

在下面的代码中，我们添加了一个名为 `focusedField` 的 FocusState 属性包装器，这是一种特殊类型的状态，会根据当前焦点位置而变化。我们可以为 FocusState 使用字符串、整数或任何其他可散列值类型，在例子中我们定义为 email 和 password 的枚举类型。初始状态下，`focusedField` 表示不聚焦于任何可聚焦项目。接下来我们使用焦点修饰符创建焦点位置和 `focusedField` 属性之间创建一个链接。这个链接是双向的，我们可以根据 `focusedField` 的值来判断并在程序中做出需要的动作。

```swift

enum Field: Hashable {
    case email
    case password
}

struct ContentView: View {

    @FocusState private var focusedField: Field?

    var body: some View {
            // 焦点与 Focus State 建立链接
          TextField("Email", text: $email)
              .focused($focusedField, equals: .email)

          SecureField("Password", text: $password)
              .focused($focusedField, equals: .password)
    }
}
```

得益于焦点与 FocusState 之间的双向链接，我们可以通过改变 `foucusField` 来移动焦点。例如，如果我们以编程方式将 focusField 的值设置为 .password，SwiftUI 将知道我们的 SecureField 与我们正在设置的新值相关联，并且焦点将自动移动到密码字段。
在示例应用程序中，当用户提交他们的数据时，我们会对其进行验证。如果电子邮件不符合预期格式，我们将 focusedField 设置为 email。这会将焦点改变回电子邮件文本字段。

```swift
enum Field: Hashable {
    case email
    case password
}

struct ContentView: View {

    @FocusState private var focusedField: Field?
    @State private var email: String = ""
    @State private var password: String = ""
    @State private var submittedEmail: String = ""

    var body: some View {
        ZStack {
                TextField("Email", text: $email)
                    .focused($focusedField, equals: .email)

                SecureField("Password", text: $password)
                    .focused($focusedField, equals: .password)
            }
            .onSubmit {
                submittedEmail = email
                if !isEmailValid {
                    focusedField = .email
                }
            }
        }
    }
  
    private var isEmailValid : Bool {
        let regex = "[A-Z0-9a-z._%+-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,64}"
        let predicate = NSPredicate(format:"SELF MATCHES %@", regex)
        return submittedEmail.isEmpty || predicate.evaluate(with: submittedEmail)
    }

}
```

通过示例程序，我们可以在 SwiftUI 中控制文本字段之间的焦点。但是，FocusState 不仅仅用于文本字段。
它们可用于以编程方式控制 iOS、tvOS、watchOS 或 macOS 上任何可聚焦视图的焦点！

**完整示例程序**

```swift
import SwiftUI
import AuthenticationServices

enum Field: Hashable {
    case email
    case password
}

struct ContentView: View {

    @FocusState private var focusedField: Field?
    @State private var email: String = ""
    @State private var password: String = ""
    @State private var submittedEmail: String = ""

    var body: some View {
        ZStack {
            Image("backgroundImage")
                .resizable()
                .opacity(0.7)
                .ignoresSafeArea()

            VStack(alignment: .center) {
                Text("Vacation Planner")
                    .font(.custom("Baskerville-SemiBoldItalic", size: 60))
                    .foregroundColor(.black.opacity(0.8))
                    .frame(alignment: .top)

                Spacer(minLength: 30)

                TextField("Email", text: $email)
                    .submitLabel(.next)
                    .textContentType(.emailAddress)
                    .keyboardType(.emailAddress)
                    .padding()
                    .frame(height: 50)
                    .background(Color.white.opacity(0.9))
                    .cornerRadius(15)
                    .padding(10)
                    .focused($focusedField, equals: .email)
                    .border(Color.red,
                            width: (focusedField == .email &&
                                    !isEmailValid) ? 2 : 0)

                SecureField("Password", text: $password)
                    .submitLabel(.go)
                    .padding()
                    .frame(height:50)
                    .textContentType(.password)
                    .background(Color.white.opacity(0.9))
                    .cornerRadius(15)
                    .padding(10)
                    .focused($focusedField, equals: .password)

                Spacer().frame(height: 20)

                HStack {
                    Rectangle().frame(height: 1)
                    Text("or").bold().padding()
                    Rectangle().frame(height: 1)

                if !isEmailValid {
                    focusedField = .email

                    // Show progress indicator, and log in.

        }
    }
  
    private var isEmailValid : Bool {
        let regex = "[A-Z0-9a-z._%+-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,64}"
        let predicate = NSPredicate(format:"SELF MATCHES %@", regex)
        return submittedEmail.isEmpty || predicate.evaluate(with: submittedEmail)
    }
```
