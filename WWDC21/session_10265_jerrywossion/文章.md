# WWDC21 10265 - 将你的应用沉浸在空间音频中

Apple 在去年的 WWDC 中推出了空间音频功能，这是一项革新的技术，将影院级的环绕声聆听体验带到了移动设备上，而且空间音频支持头部追踪技术，可以使声场聚焦在播放设备上而不随头部的转动而变化。然而最初只有 AirPods Pro 和 AirPods Max 支持空间音频，而且只在特定格式的音视频资源（如杜比全景声）上才能生效，同时也没有提供相关 API 供开发者使用。今年的 WWDC 上，Apple 则比较详细介绍了空间音频以及相关 API，新增了支持的设备和音频格式，同时也可以通过 Up-Mix 黑科技让普通的立体声资源也可以提供空间音频效果。

本文通过四个部分来介绍这些内容：
1. 空间音频和头部追踪技术简介；
2. 如何在提供空间音频功能时自适应用户带宽；
3. Apple 提供的相关 API；
4. 空间音频在 Apple 设备和操作系统版本间的可用性。

## 空间音频是什么

### 立体声的不足

![Stereo](images/stereo.png)

在介绍空间音频之前，先回顾一下普通的立体声技术，立体声在聆听体验方面有两个缺陷：

1. 立体声一般使用两个声道来构造立体感的声音，但是由于缺少位置信息，我们感知到的音场（soundstage）是极其受限的，不管是使用耳机还是立体声扬声器，我们无法听到背后的声音、正前方的声音、头上的声音等，因此立体声难以营造栩栩如生的环绕声体验。

2. 当我们佩戴耳机时，声音从耳机里的微型扬声器中发出，声源位置相对于我们的头部是固定的，当我们移动头部时，这些扬声器也随着移动，听起来感觉声音的位置一直在变化，这种聆听体验叫做“头上体验（in-head experience）”，和我们在影院、在现实中聆听声音的体验是不同的，在这些场景中，即时我们转动头部，听起来声源的位置是固定的。

### 空间音频环绕声

![头外体验](images/out-of-head.png)

空间音频是一种可以带来环绕声听觉体验的技术，环绕声基于[心理声学（psychoacoustic）技术](https://en.wikipedia.org/wiki/Psychoacoustics)，利用人耳[音源定位](https://en.wikipedia.org/wiki/Sound_localization)的原理，通过细微地改变左右耳接收到的音频参数（响度、频率、时域等），可以营造出引人入胜的虚拟空间声场。利用空间音频技术，开发者可以提供如同置身于音乐会之中的环绕的音乐体验，也可以打造带有交互场景的全动态视频游戏，给玩家带来沉浸式的冒险体验。
### 空间音频头外体验

和立体声的头上体验不同，当使用空间音频时，我们感受到的虚拟声场是静态的，不会随着我们头部的移动而移动。当使用支持空间音频的耳机时，播放设备中的惯性测量单元（inertial measurement units）会测量头部当前的移动状态信息，用于确定听者的头部姿势，然后根据用户头部姿势的变化，播放设备会动态地调整音频渲染输出，从而维持一种静态声场效果，这种体验叫做头外体验（out-of-head experience），会使人听起来感觉声源的位置是固定的，从而更加自然真实，生动丰富。即使在转弯的公交车上，或者在倾斜的飞机上，空间音频效果依然保持。

## 空间音频带宽自适应

在提供空间音频服务时，必须面对的一个问题是用户的带宽是有限的，而且不稳定。支持环绕声的音频资源一般有多个声道，常见的有如 5.1 声道、7.1 声道、杜比全景声等，空间音频对这些多通道音频资源的支持效果最好。但是由于声道变多、比特率变高，多通道音频资源文件的体积也大很多，在传输时将占用更多的带宽。如果提供的是视频服务，在有限的带宽环境下多声道音频还会压缩画质，造成不好的视听体验。那么有没有既能提供空间音频效果，又能避免尺寸过大的问题呢？

好消息是，Apple 利用 Up-Mix 技术可以将立体声音频资源渲染成类似于 5.1 声道的音频效果，相比之下立体声资源的文件体积小很多，而且许多应用可能并没有多声道的音频资源，而是有更普遍的立体声资源，有了 Up-Mix 技术，应用可以用现有的立体声音频资源库来为用户提供空间音频，据 Apple 介绍，Up-Mix 后的效果也很出色。

![up-mix](images/up-mix.png)

Apple 推荐使用 HLS 协议来分发音频资源。HLS 是一种流式协议，具有很好的带宽适应能力，可以分发不同版本的音频资源来适应不同的带宽情况。基于 HLS 和 Up-Mix 技术，开发者可以提供带宽自适应的空间音频服务：

- 当用户的带宽情况比较好时，可以直接下发多通道音频资源，为用户带来最佳的聆听体验。此外，Apple 原生支持多声道音频资源空间音频，直接下发多通道音频资源的话不需要在软件层面做任何改动；
![high-bandwidth](images/high-bandwidth.png)
- 当用户的带宽不足以提供高质量的试听体验时，音频会无缝地降级为立体声资源，并使用 up-mix 技术提供空间音频处理。如果在降级转换之前提供了头部追踪，会继续维持追踪效果。之后，如果带宽恢复，完整的多声道空间音频处理也会恢复。
![low-bandwidth](images/low-bandwidth.png)

### 响度均衡

为了自适应用户带宽，HLS 协议会下发不同版本的音频资源，AAC、xHE-AAC、Dolby Digital Plus 等 codec 的 metadata 中包含响度均衡的一些重要参数，比如 Dolby E 中的 `dialnorm` 用于决定解码器输出的音量值，xHE-AAC 中的 [DRC Metadata](https://developer.apple.com/documentation/http_live_streaming/providing_metadata_for_xhe-aac_video_soundtracks) 用于动态调整音频输出幅度，在音量比较大的时候压制音量，音量小的时候适度提高音量。Apple 建议在分发音频资源的时候提供适当的参数值，以做好这些不同版本资源间的响度均衡，达到无缝切换的效果。具体参见 Apple 官网的[相关文档](https://developer.apple.com/documentation/http_live_streaming/hls_authoring_specification_for_apple_devices)。

## API

最终是否启用空间音频取决于多个部分：
1. 开发者决定自己的音频内容是否支持空间音频
2. 播放设备是否支持空间音频
3. 用户是否在控制中心或蓝牙设置中启用了空间音频

Apple 提供了若干 API 来针对这几个方面进行设置、查询或监听。
### 设置应用允许的空间音频资源格式

前文提到空间音频原生支持多声道音频，也可以通过 Up-Mix 支持立体声音频，开发者通过 `AVPlayerItem` 或者 `AVSampleBufferAudioRenderer` 中的 `allowedAudioSpatializationFormats` 属性来配置自己的应用所允许的空间音频资源格式，具体取值如下：

```Swift
// AVPlayerItem or AVSampleBufferAudioRenderer
@available(macOS 11.0, *)
var allowedAudioSpatializationFormats: Int32
```

其中 `AVAudioSpatializationFormats` 取值如下：

```Swift
public struct AVAudioSpatializationFormats : OptionSet {
    /// rawValue 传 0 时禁用空间音频
    public init(rawValue: UInt)
    /// 只对单声道或立体声格式音频进行空间音频化
    public static var monoAndStereo: AVAudioSpatializationFormats { get }
    /// 只对多声道格式音频进行空间音频化
    public static var multichannel: AVAudioSpatializationFormats { get }
    /// 以上二者均进行空间音频化
    public static var monoStereoAndMultichannel: AVAudioSpatializationFormats { get }
}
```

### 向系统宣告空间音频支持

> 注：这个 API 官方文档中目前尚无具体说明，这里是笔者对 WWDC Session 中只言片语的理解

在 `AVAudioSession` 中，开发者可以通过 `setSupportsMultichannelContent` 方法向系统宣告你的应用支持多声道空间音频，如果用户没有在控制中心或蓝牙设置中启用空间音频选项，在播放多声道内容时系统会显示一个提示（目前 Apple 并没有提供具体的信息这个提示以何种形式呈现）。如果你的应用使用的是 `AVPlayer` ，系统会帮你管理这个提示。

```Swift
extension AVAudioSession {
    @available(iOS 15.0, *)
    func setSupportsMultichannelContent(_ inValue: Bool) throws
    @available(iOS 15.0, *)
    var supportsMultichannelContent: Bool { get }
}
```

### 空间音频启用查询与变更监听

通过参数 `isSpatialAudioEnabled` 来判断播放设备（音频路由：audio route）是否支持空间音频：

```Swift
@available(iOS 6.0, *)
class AVAudioSessionPortDescription : NSObject {

    @available(iOS 15.0, *)
    var isSpatialAudioEnabled: Bool { get }

}
```

音频路由发生变化时，audio session 会发出 `AVAudioSession.routeChangeNotification` 通知，开发者可以监听该通知，检查 `isSpatialAudioEnabled` 属性，然后做出相应的处理。

当用户改变控制中心或蓝牙设置中的空间音频偏好时，`AVAudioSession` 会发出一个 `spatialPlaybackCapabilitiesChangedNotification`，开发者可以通过 `AVAudioSessionSpatialAudioEnabledKey` 来获取该通知携带的关于空间音频启用状态的信息：

```Swift
extension AVAudioSession {
    @available(iOS 15.0, *)
    class let spatialPlaybackCapabilitiesChangedNotification: NSNotification.Name
}

@available(iOS 15.0, *)
let AVAudioSessionSpatialAudioEnabledKey: String
```

## 功能可用性

最后我们看一下空间音频的可用性，从硬件和系统两个维度来看一下哪些设备支持空间音频。

### 首先具备空间音频播放能力的播放设备有：
- 这些设备的内置扬声器：2018 年之后的 MacBook，iPhone，iPad Pro，2020 款 iPad Pro，2021 款 iMac
- AirPods Pro 和 AirPods Max

搭配 AirPods Pro/AirPods Max，可以在这些本身不支持空间音频播放能力的设备上体验空间音频：
- 2016 年之后的 iPhone 和 iPad

其中如果要启用 AirPods Pro/AirPods Max 的空间音频能力，和它们连接的设备系统版本至少要是 macOS Big Sur、iOS 14 和 iPadOS 14 及以上。

> 奇怪的是，本次 WWDC 上，Apple 并没有把 Apple Silicon Mac 归到内置扬声器支持空间音频的分类里，而是分到了需要搭配 AirPods Pro/AirPods Max 使用的类别里，并且备注只支持带视频的音频（audiovisual），官方几乎找不到对这点的说明，需要后续关注跟进。

### 系统版本维度空间音频功能可用性如下：

在 macOS Catalina、iOS 13 和 iPad OS 13 这些系统下，支持 2018 年及以后年份型号的 MacBook、iPhone和 iPad Pro，通过内置的扬声器对下列音频播放方式提供空间音频支持：
- `AVPlayerItem`
- WebKit `<video>` 标签指定的任意 http scheme

默认情况下，选择可用的多声道音频进行空间音频化。

![13](images/13.png)

在 macOS Big Sur、iOS 14 和 iPadOS 14 系统中，Apple 添加了对 AirPods Pro 和 AirPods Max 耳机的支持，利用这些配件，可以为 2016 年及以后的 iPhone 和 iPad 设备提供空间音频支持。

默认情况下仍然是选择可用的多声道音频进行空间音频化。

![14](images/14.png)

最后，在最新的 macOS Monterey、iOS 15、iPadOS 15 以及 tvOS 15 中，Apple 对下列音频播放方式提供空间音频支持：
- `AVPlayerItem`
- `AVSampleBufferAudioRenderer`
- WebKit W3C MSE 扩展

其中 MSE 没有对应的 API 支持，不过可以通过 Media Capabilities API 在 AudioConfiguration 字典中查询空间音频支持的可用性。

默认情况下在可用且条件允许的情况下为所有的单声道、立体声和多通道音源提供空间音频化。对于 `AVSampleBufferAudioRenderer` 只含有音频的播放场景，默认情况下只提供多通道音频的空间音频化。
